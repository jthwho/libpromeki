Detailed Description

A built-up HTTP response.

HttpResponse is the half of the request/response pair that server handlers populate (and that clients receive). It carries a status, a header collection, and a body.

The body has two backing forms, used in the obvious order of preference:

  1. An in-memory Buffer (set via setBody, setText, setJson, setHtml, setBinary). Cheap, simple, and the right call for the vast majority of API responses.
  2. A streamed IODevice (set via setBodyStream). Used by HttpFileHandler so the response can pour out a multi-GB file without loading it all at once. When a stream is set, the in-memory body is ignored.

Convenience factories (ok, notFound, badRequest, internalError) cover the common quick-reply paths.

Thread Safety
Distinct instances may be used concurrently — copies share the underlying body Buffer via atomic refcount. A single instance is conditionally thread-safe: const accessors are safe, mutators (setStatus, setBody, setHeader, ...) require external synchronization.
Example
HttpResponse res = HttpResponse::ok(makeJsonObject());
res.setHeader("Cache-Control", "no-store");
// Or, hand-built:
HttpResponse r;
r.setStatus(HttpStatus::Created);
r.setHeader("Location", String::sprintf("/items/%d", id));
r.setJson(itemAsJson);

◆ UpgradeHook

using promeki::HttpResponse::UpgradeHook = std::function<void(TcpSocket *socket)>

Callback invoked after a 101 response finishes writing.

Used by HttpServer::routeWebSocket to detach the underlying socket from the HTTP layer once the "Switching Protocols" response has drained. The callback receives the open, non-blocking TcpSocket (or SslSocket); ownership transfers to the callback, and the originating HttpConnection self-destructs immediately afterwards.

Set on a 101-status response by application or library handlers; ignored on non-101 responses.

Reported length of a streamed body, or -1 if unknown.

When known, the connection writes a Content-Length header and pours the device byte-for-byte. When unknown (size() returned -1), the connection falls back to Transfer-Encoding: chunked.

Streams a body from an IODevice.

Takes ownership of device. When length is non-negative, sets Content-Length and writes exactly that many bytes; when -1, switches to Transfer-Encoding: chunked and writes until the device's read returns 0.

If mimeType is non-empty, sets Content-Type to it. Otherwise the caller is responsible for setting the header.
