Detailed Description

RTP session for sending and receiving packets (RFC 3550).

RtpSession manages the RTP protocol state for a single synchronization source (SSRC). It handles RTP header construction, sequence number management, timestamp tracking, and packet transmission via a PacketTransport.

The session operates on pre-built RtpPacket lists from RtpPayload handlers: it fills in the RTP header fields (version, payload type, sequence number, timestamp, SSRC, marker bit) and hands the packets to the transport for delivery.

Transport ownership

RtpSession can either own its transport (when started via start(const SocketAddress&), which creates an internal UdpSocketTransport) or borrow a caller-owned transport (when started via start(PacketTransport*)). Borrowing is the right pattern when the caller needs to configure the transport with options RtpSession doesn't expose, or when the same transport is being shared across multiple RTP streams (not currently supported, but forward-compatible with future DPDK multiplexing).

Destination

The destination address is set once via setRemote() before sending any packets. RTP streams have a single peer (unicast address or multicast group), so per-send destination arguments would only invite inconsistency. To send to a multicast group, pass the group address to setRemote() and configure the transport (TTL, outgoing interface) before start().

Example
RtpSession session;
session.setPayloadType(96);
session.setClockRate(90000);
session.setRemote(SocketAddress(Ipv4Address(239, 0, 0, 1), 5004));
session.start(SocketAddress::any(0));
RtpPayloadJpeg payload(1920, 1080);
auto packets = payload.pack(jpegData, jpegSize);
session.sendPackets(packets, timestamp, true);

Sends a list of packets with per-packet timestamps.

Each packet is stamped with startTimestamp + i * timestampStride so consecutive packets carry monotonically advancing RTP timestamps. This is the AES67 / ST 2110-30 audio path, where every packet-time interval gets its own timestamp reflecting the exact sample index at the start of that packet.

Sequence numbers still come from the session's monotonic counter and the marker bit is applied uniformly (audio streams should pass marker = false). Packets are handed to the transport as a single batch via PacketTransport::sendPackets().

Parameters
packetsThe pre-packed packet list, modified in-place.
startTimestampThe RTP timestamp for the first packet's first sample.
timestampStrideIncrement applied per successive packet.
markerMarker bit applied to every packet.
Returns
Error::Ok on success, or an error on failure.

◆ startReceiving()

Error RtpSession::startReceiving ( PacketCallback  callback,
const String &  threadName = "rtp-rx" 
)

Starts the per-session RTP receive loop.

Spawns an internal Thread named "rtp-rx" that calls PacketTransport::receivePacket in a loop, wraps each datagram in an RtpPacket view of an owned Buffer, and delivers the packet to callback. The existing packetReceived signal is also emitted for consumers that prefer event-loop dispatch.

The session must already be running (isRunning() == true) — call one of the start overloads first.

The receive thread polls the stop flag every few hundred milliseconds using SO_RCVTIMEO on the underlying socket, so stopReceiving() returns within one poll interval. If the transport is not a UDP socket the poll timeout is a best-effort — transports that cannot expose a non-blocking receive may stall stopReceiving until a packet arrives.

Parameters
callbackThe per-packet callback (required).
threadNameThread name surfaced to debuggers and OS monitors. Defaults to "rtp-rx".
Returns
Error::Ok on success, Error::Busy if already receiving, Error::NotOpen if the session has not been started, or Error::InvalidArgument on a null callback.
