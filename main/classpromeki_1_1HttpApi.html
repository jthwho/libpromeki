Detailed Description

Self-describing HTTP API surface with built-in catalog and explorer.

HttpApi sits beside an HttpServer and tracks a registry of Endpoint descriptors. Every endpoint registered through the API is mounted on the server and recorded in the catalog, so the two never drift out of sync. The catalog is exposed three ways:

  • GET <apiPrefix>/ — bundled HTML/JS explorer served from :/.PROMEKI/explorer/, which fetches /_openapi at load time and renders a per-endpoint form UI. The explorer lives at the api's bare prefix (Swagger-style) so it's discoverable without knowing any internal path.
  • GET <apiPrefix>/_catalog — promeki-native JSON list of every registered endpoint.
  • GET <apiPrefix>/_openapi — OpenAPI 3.1 document covering the same endpoints (validator-clean, importable into Swagger UI / Redoc / Postman / openapi-generator).
Thread Safety
Inherits ObjectBase — thread-affine. Construction captures the HttpServer reference; both objects must share a single EventLoop (typically the main loop). Endpoints registered after mount are visible immediately — the catalog is generated on demand on each request.
Decoupling from HttpServer
The server has no compile-time dependency on HttpApi — the existing HttpServer::exposeDatabase / HttpServer::exposeLookup helpers stay available for "internal mount, don't advertise" cases. HttpApi provides matching exposeDatabase / exposeLookup helpers that delegate to the server and publish catalog entries — that's the recommended path so the explorer reflects the live API.
Multiple APIs on one server
Because the server is unaware of the API registry, multiple HttpApi instances can co-exist on a single HttpServer at different prefixes (e.g. /api/public and /api/admin), each with its own catalog and explorer.
Example
Application app(argc, argv);
HttpServer server;
HttpApi api(server, "/api"); // every endpoint nests under /api
// Endpoint paths are RELATIVE to the api's prefix — the line below
// registers a route at /api/health on the underlying server.
api.route(HttpApi::Endpoint{
.path = "/health",
.method = HttpMethod::Get,
.title = "Health check",
.summary = "Returns 200 when the service is responsive.",
.tags = {"system"},
.response = VariantSpec().setType(Variant::TypeString),
},
[](const auto &, auto &res) { res.setText("ok"); });
api.exposeDatabase("/options", "Library options",
LibraryOptions::db(), true);
api.installPromekiAPI(); // optional: adds /api/promeki/{build,env,log,...}
api.mount(); // adds /api/ (explorer UI), /api/_catalog, /api/_openapi
server.listen(8080);
return app.exec();

◆ RpcCall

using promeki::HttpApi::RpcCall = std::function<Result<Variant>(const VariantMap &args)>

RPC-style handler: typed args in, Variant out.

Each declared Param from the Endpoint shows up in the VariantMap keyed by its name — already coerced into the type its VariantSpec specified, with range and enum constraints validated. Missing optional params either appear at their declared default or are absent entirely; missing required params cause the call to be rejected with 400 before the handler runs.

Returning a successful Result with a Variant produces a 200 with {"value": <variant>} as the body (serialized via JsonObject::setFromVariant). Returning a populated Error maps to one of the declared ErrorResponse entries by status code, falling back to 500 if none matches.

◆ installPromekiAPI()

Error promeki::HttpApi::installPromekiAPI ( )

Installs the built-in promeki API surface.

Mounts every diagnostic the library ships with — build info, environment snapshot, library options, memory-space stats, logger control, plus the static debug UI — under <prefix>/promeki/, where each module appears as a sibling endpoint (e.g. /promeki/build, /promeki/log) alongside the UI's index.html at the bare path.

Intentionally all-or-nothing: applications that want fine-grained control over which diagnostics they expose should not call this method and instead register their own bespoke endpoints. The bundled debug UI assumes the entire surface is present.

Safe to call exactly once before mount.

Returns
Error::Ok or Error::Exists if any endpoint collides with an existing registration.

◆ mount()

Mounts the catalog, OpenAPI, and explorer routes.

Installs three routes under prefix (see class docs). Safe to call exactly once before HttpServer::listen — calling it twice registers duplicate routes and returns Error::Exists on the second call.

ep is moved into the catalog, then routed on the underlying server. The handler runs with the raw request — no parameter unmarshalling — so this is the right entry point when the handler needs full control over the response format (e.g. streaming, non-JSON content, custom error shapes).

◆ rpc()

Error promeki::HttpApi::rpc ( Endpoint  ep,
RpcCall  call 
)

Registers an RPC-style endpoint.

The framework parses each declared Param from its declared ParamIn, validates against its VariantSpec, and hands the resulting VariantMap to call. A successful Result is rendered as {"value": <variant>} JSON with status 200; an error Result picks a matching ErrorResponse by status code (falling back to 500).

The default method is HttpMethod::Post when ep.params contains any ParamIn::Body entries, otherwise whatever ep.method already specifies.

◆ toOpenApi()

JsonObject promeki::HttpApi::toOpenApi ( ) const

Renders an OpenAPI 3.1 document for every registered endpoint.

Maps VariantSpec to JSON Schema using the conversion table in variantSpecToJsonSchema — domain types (Timecode, UUID, PixelFormat, …) become "string" with a "format" extension, plus a "components.schemas" entry for any complex shapes.

The result is validator-clean against the official OpenAPI 3.1 JSON Schema; CI is expected to lint it with openapi-spec-validator or equivalent.
