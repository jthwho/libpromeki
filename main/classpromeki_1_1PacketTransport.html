Detailed Description

Abstract packet-I/O transport for datagram-oriented protocols.

PacketTransport decouples higher-level packet protocols (RtpSession, future RTCP, SRT, etc.) from the concrete kernel or userspace path that actually moves bytes. It is the layer where kernel socket vs. DPDK vs. AF_XDP vs. in-process loopback is chosen, so that the RTP and MediaIO layers never talk to sockets directly and can be reused unchanged across transport backends.

The interface is deliberately narrow:

  • open() / close() acquire and release whatever the transport needs (a kernel socket, a DPDK port, a loopback queue).
  • sendPacket() and sendPackets() move outbound datagrams to one or many destinations. The batch form is the primary API because DPDK and sendmmsg() both prefer to see many packets at once; the single-packet form exists for convenience.
  • receivePacket() reads one inbound datagram.
  • setPacingRate() and setTxTime() expose optional transmit-rate and per-packet deadline controls. Backends that cannot implement them return Error::NotSupported.
Lifetime

Transports are owned by the caller. They are not reference counted — an RtpSession or MediaIO backend that takes a transport pointer does not own it; the caller is responsible for keeping it alive until the session is stopped.

Thread Safety
Conditionally thread-safe. Distinct transport instances may be used concurrently. A single instance must only be used from one thread at a time — like the underlying socket, no internal locking is performed on send / receive paths.
Buffer lifetime

When sendPackets() accepts a Datagram list, the backing data pointers must remain valid until the call returns. Socket backends copy into kernel buffers synchronously, so this contract is free for the caller; DPDK backends must copy or refcount before returning, not defer the copy.

◆ sendPackets()

virtual int promeki::PacketTransport::sendPackets ( const DatagramList &  datagrams)
pure virtual

Sends a batch of packets.

Backends implement this as sendmmsg() on Linux sockets, as a ring-burst on DPDK, or as a queue enqueue on loopback. The call returns the number of datagrams the transport accepted before erroring; a fully successful call returns datagrams.size().

Parameters
datagramsThe list of datagrams to send.
Returns
The count of accepted datagrams, or -1 if none were sent.

Implemented in promeki::LoopbackTransport, and promeki::UdpSocketTransport.
