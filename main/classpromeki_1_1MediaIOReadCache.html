Detailed Description

Producer-side prefetch buffer for MediaIOSource reads.

A MediaIOReadCache owns the "keep N reads in flight" policy for a single source. It maintains a queue of MediaIOCommandRead shared pointers — some in-flight on the strand, some already completed and waiting to be vended — and serves one read per MediaIOSource::readFrame call by handing back a request bound to the head of that queue.

Invariants
  • The queue holds at most depth() commands at any time.
  • Vended requests stay valid forever (they share ownership of the cmd via MediaIOCommand::Ptr). Dropping a not-yet-completed request silently discards the produced frame; the strand still runs the cmd to completion.
  • After every readFrame call the cache tops up to depth() so the prefetch invariant is restored before returning.
frameReady semantics
MediaIOSource::frameReady fires on the transition "head of the cache becomes ready." Concretely, the cache emits the signal exactly once between any pair of readFrame pops; once a head is "armed" it stays armed until readFrame consumes it (at which point the cache re-checks the new head and re-arms if it is already complete). Polling consumers can therefore wait on frameReady and trust that a subsequent readFrame will resolve without blocking.
Threading
Public methods are safe to call from any thread. Internally a Mutex serializes queue mutation; onCommandCompleted is called from the strand worker via MediaIO::completeCommand, while readFrame is called from the user thread.
Lifetime
The cache is owned by its MediaIOSource as a value member, so the back-pointer to the source is stable for the life of the cache. Callers must drain the strand (e.g. by closing the MediaIO) before destroying the source — the MediaIO destructor handles this with _strand.waitForIdle().

Drains the queue and cancels every held cmd.

Marks every queued cmd cancelled (so the strand worker short-circuits if it has not yet started) and clears the queue. Already-vended requests are unaffected — their underlying cmd still completes (with Error::Cancelled when the cancel raced dispatch) and the request resolves normally.

◆ onCommandCompleted()

void promeki::MediaIOReadCache::onCommandCompleted ( )

Notifies the cache that some held cmd just completed.

Called from MediaIO::completeCommand on the strand worker thread after the per-Read cache writes have run but before the cmd's MediaIOCommand::markCompleted fires. Drives the MediaIOSource::frameReady edge-detection.

Safe to call when the completed cmd is no longer in the queue (for example, after cancelAll dropped it but before the strand worker noticed). In that case the cache simply re-evaluates the armed flag against the current head.

◆ pushSyntheticResult()

void promeki::MediaIOReadCache::pushSyntheticResult ( Error  err)

Pushes an already-completed synthetic read result.

Used by MediaIO::completeCommand on the close path to deliver a trailing EOS to consumers. The synthetic cmd carries err in its MediaIOCommand::result and is marked completed so readFrame returns it as an immediately- resolved request. The frameReady armed flag is re-evaluated after the push so polling consumers observe the synthetic result through the same mechanism as a live read.

◆ readFrame()

MediaIORequest promeki::MediaIOReadCache::readFrame ( )

Vends the next read.

Returns a request bound to the head of the queue. If the queue is empty, submits a fresh MediaIOCommandRead first. Tops up the queue to depth before returning so the next caller sees a primed prefetch.

The returned request's wait will block when the head was still in flight (and resolves when the strand worker completes the cmd) or fire immediately when the head was already complete (cache hit). Either way the cmd's typed payload (MediaIOCommandRead::frame, currentFrame, descriptor-change flags) is reachable via req.commandAs<MediaIOCommandRead>() once the request resolves.

◆ setDepth()

void promeki::MediaIOReadCache::setDepth ( int  n)

Sets the target prefetch depth.

Clamped to a minimum of 1. The new depth applies on the next readFrame top-up; the cache does not retroactively shrink an over-full queue (which can happen briefly if the depth is lowered while reads are in flight).
