Detailed Description

Sequential IODevice that ferries shared Buffer segments from a producer to a consumer with async-read semantics.

AsyncBufferQueue holds a thread-safe queue of Buffer::Ptr segments alongside a "writer side closed" latch. The producer enqueues Buffer::Ptr segments via enqueue and signals end-of-stream via closeWriting. The consumer drains via read on whichever thread owns the device.

The device is purposely shaped so that a parked HTTP body writer can tell "no bytes yet, wake me later" apart from "stream is finished":

  • When the queue is empty and closeWriting has not been called, read returns 0 and atEnd returns false. The consumer should park on readyReadSignal — every enqueue emits it from the producer's thread.
  • When the queue is empty and closeWriting has been called, read returns 0 and atEnd returns true. This is the standard end-of-stream condition.

Threading:

  • enqueue and closeWriting are safe to call from any thread. Both emit readyReadSignal so a consumer hosted on an EventLoop different from the producer will be marshalled awake by the existing cross-thread connect machinery.
  • read, atEnd, bytesAvailable, open, and close should be called only from the consumer's thread.
Example
AsyncBufferQueue queue;
queue.open(IODevice::ReadOnly);
queue.readyReadSignal.connect([&]() {
char tmp[1024];
int64_t got = queue.read(tmp, sizeof(tmp));
if(got > 0) consume(tmp, got);
});
// From a producer thread:
queue.enqueue(jpegPtr);
queue.enqueue(boundaryPtr);
queue.closeWriting();

◆ closeWriting()

void promeki::AsyncBufferQueue::closeWriting ( )

Latches the writer side closed.

After this call enqueue returns Error::NotOpen, and atEnd flips to true once the consumer drains the remaining segments. Emits readyReadSignal one final time so a parked consumer wakes and observes the new end-of-stream condition. Idempotent.
