Detailed Description

Write-side port of a MediaIO.

A MediaIOSink accepts frames into a MediaIO instance. It carries the write API that used to live on MediaIO directly — the per-port writeFrame entry point, per-port format-negotiation surface (expectedDesc / proposeInput), and the per-port write-side signals (frameWantedSignal, writeErrorSignal).

Sinks are created by the backend during open and parented to their MediaIOPortGroup, which is in turn parented to the owning MediaIO. Per-type indexing is independent of source ports — a single MediaIO may simultaneously expose sink(0) and source(0) as distinct ports.

Default name
If the name argument is empty the constructor synthesizes "sink{index}" (e.g. "sink0", "sink1") so every sink has a stable, human-readable identifier even when the backend has nothing specific to call it.
Port groups
Sinks placed in a multi-sink MediaIOPortGroup are dispatched in lockstep: a sync-grouped write is held on the strand until every sink in the group has a frame queued for the same group position, then the group's frames go to the backend together as one MediaIOCommandWrite. Sinks in single-port groups behave independently. Pending-write counting lives on the group, since a single command advances the whole group at once; pendingWrites is a per-sink convenience that forwards to the group's count.

◆ writeFrame()

MediaIORequest promeki::MediaIOSink::writeFrame ( const Frame::Ptr &  frame)

Submits a frame to the sink.

Builds a MediaIOCommandWrite, dispatches it through the owning MediaIO's submit gate, and returns a request the caller can .wait() on or attach a .then() continuation to. Per the always-async API rule callers that want synchronous behavior write sink->writeFrame(f).wait().

Capacity gate
When writesAccepted reports zero the request resolves immediately with Error::TryAgain without queueing anything. Callers wait for the frameWantedSignal before retrying.
Pre-open / closed
Resolves with Error::NotOpen when the owning MediaIO is not open or is in the closing state, or Error::Invalid when the sink is detached.
Parameters
frameThe frame to write. May be invalid for flush-only commands.
Returns
A request resolving with the per-write outcome. The same writeError signal as before still fires for back-compat with signal-driven consumers — request callers can ignore it and read the resolution from the request instead.

◆ writesAccepted()

int promeki::MediaIOSink::writesAccepted ( ) const

Returns the number of additional writes the sink will accept now.

Computed from writeDepth() minus the in-flight pending count and the backend task's own internal output queue depth. Returns 0 when no further writes will be accepted without blocking.

◆ writeErrorSignal

Signal< Error > promeki::MediaIOSink::writeErrorSignal

Emitted when an asynchronous write resolves with an error.

Signal:

Per the always-async API, every writeFrame returns immediately and the actual write runs on the strand worker thread. Callers that hold the returned MediaIORequest learn about errors directly from its resolution; this signal is the fan-out path for fire-and-forget consumers and for downstream observers ( MediaIOPortConnection, pipeline status panes ) that don't track individual requests.
