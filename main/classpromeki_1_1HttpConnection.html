Detailed Description

Per-connection HTTP/1.x request parser, response serializer, and socket pump.

One HttpConnection wraps one accepted TcpSocket. The connection is the bridge between byte-oriented I/O and the higher-level HttpRequest / HttpResponse types: it feeds received bytes into a vendored llhttp parser, builds a request object as the parser fires its callbacks, hands the request to a caller-supplied dispatcher when the message is complete, then serializes the produced HttpResponse back out.

Lifecycle expectations:

  • Construct with the accepted TcpSocket (ownership is transferred).
  • Call start to register the socket fd with the connection's EventLoop and begin reading.
  • The connection emits requestReceived / responseSent / closed signals on the loop's thread; the dispatcher callback installed via setRequestHandler is invoked on the loop thread between request completion and response start.

Keep-alive: HTTP/1.1 keeps the connection alive by default; HTTP/1.0 closes by default. Either side's Connection: close header forces close after the in-flight response is sent.

Streamed bodies: when the response carries an HttpResponse::bodyStream, the connection drains the device incrementally as the socket becomes writable, falling back to chunked transfer-encoding when the device's length is unknown.

Note
HttpConnection is internal infrastructure. Application code should use HttpServer; the connection is exposed as a public type only so out-of-tree integrations and tests can drive it directly when needed.
Thread Safety
Inherits ObjectBase — thread-affine. All connection state mutates on the EventLoop thread that owns the underlying TcpSocket. Cross-thread interaction goes through the loop's postCallable.

Posts a response back through this connection.

Used by handlers that defer their reply onto a worker thread: copy the request, do the work, then call this method from the connection's EventLoop thread (typically via postCallable).

Safe to call exactly once per requestReceived emission; calling more than once raises an error.

◆ setNeedsServerHandshake()

void promeki::HttpConnection::setNeedsServerHandshake ( )

Tells the connection to drive a server-side TLS handshake before reading the first request.

Called by HttpServer immediately after construction when the wrapped socket is an SslSocket configured with a server certificate. The handshake runs on the loop's I/O readiness events; the parser only starts accepting bytes once the handshake completes successfully.

Must be called before start. No-op when PROMEKI_ENABLE_TLS is off.

◆ setRequestHandler()

Installs the request dispatcher.

The dispatcher is what hands off the parsed HttpRequest to the application's routing logic (typically HttpServer::router). Without a dispatcher, completed requests immediately respond with HttpStatus::NotImplemented.
