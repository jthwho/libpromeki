Detailed Description

Reusable scheduling helper that paces a series of work items against an external Clock.

Backends that need to emit work at a steady cadence against a clock they don't own (a capture-card video clock fed forward to an NDI sender, an AES67 PTP clock fed forward to an RTP sender) embed a PacingGate per stream. Each call to wait advances the gate's internal timeline by one period (or a caller-supplied duration), sleeps if there's slack, and returns a verdict the caller acts on.

Lifecycle
  1. Construct with no clock (or pass one in).
  2. Set the clock via setClock when one becomes available (typically in the backend's executeCmd(MediaIOCommandSetClock) handler).
  3. Set the per-tick setPeriod (defaults to zero — the no-arg wait then is a no-op).
  4. Call wait once per work item. The first call after binding a clock latches an anchor against clock->now() and returns immediately so the first item ships without delay. Subsequent calls sleep against anchor + cumulative_advance.
Verdict thresholds
  • setSkipThreshold (default: one period) — once the lag exceeds this, wait returns PacingVerdict::Skip.
  • setReanchorThreshold (default: eight periods) — once the lag exceeds this, wait re-anchors and returns PacingVerdict::Reanchor. Both thresholds are absolute durations; a backend whose period changes mid-stream should re-set them after the period change.
Telemetry

Cumulative counters (ticksOnTime, ticksLate, ticksSkipped, reanchors) are exposed for backends that roll them into MediaIOStats — no per-call allocation, no locking, just integers updated on each wait.

Thread Safety

Not internally synchronized. Each PacingGate is meant to be embedded in a backend's thread-confined state — typically the dedicated worker thread of a DedicatedThreadMediaIO or the strand of a SharedThreadMediaIO. External serialization is required if multiple threads need access.

◆ tryAcquire() [2/2]

bool promeki::PacingGate::tryAcquire ( const Duration &  advance)

Non-blocking "is the next tick due?" probe.

Counterpart to wait for callers that want a rate-limiter shape rather than a pacer shape. Where wait sleeps to slow a fast pump down to the clock's cadence, tryAcquire just answers "yes, advance the timeline" or "no, you arrived early — drop / no-op." Used by sinks that simulate real playback devices: such a sink can only consume one frame per period, and an early arrival is a drop rather than something to wait for.

Behavior:

  • No clock bound → always returns true (no pacing constraint to enforce). The timeline does not advance — there is no deadline to track.
  • First call after construction / rearm / setClock → latches the anchor against clock->now() and returns true.
  • Subsequent call with now >= anchor + accumulated + advance → advances the timeline by advance and returns true. If the new lag after the advance exceeds reanchorThreshold the gate re-anchors and counts a reanchor.
  • Otherwise (now < deadline, "arrived early") → returns false; the timeline does not advance.
  • Clock now() failure → returns false (treat as "not due"); callers needing the underlying error should call wait.
Parameters
advanceThe per-tick interval.
Returns
true to consume, false to drop.

◆ wait() [2/2]

PacingResult promeki::PacingGate::wait ( const Duration &  advance)

Sleeps until the next deadline and returns a verdict.

On the first call after construction or rearm, latches anchor = clock->now() and returns immediately with OnTime / zero slack so the first work item ships without delay. Subsequent calls compute deadline = anchor + accumulated and sleep via clock->sleepUntil(deadline).

No-op when the clock is null (returns OnTime, zero slack, Error::Ok).

Parameters
advanceHow much to advance the timeline by for this work item. For fixed-rate streams, pass period (or use the no-arg overload). For variable-rate streams (e.g. audio blocks of irregular sample counts), pass the per-block duration.
Returns
PacingResult describing what the gate did and what the caller should consider doing.
