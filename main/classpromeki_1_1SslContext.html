Detailed Description

TLS/SSL configuration shared across SslSocket instances.

SslContext owns the certificate / private-key / trust-store material plus the protocol-version policy that an SslSocket uses when handshaking. One context is typically built once at startup (server side) or program init (client side) and shared by every socket.

Underneath, the context wraps mbedTLS's mbedtls_ssl_config, mbedtls_x509_crt, and mbedtls_pk_context types — the pImpl is heap-allocated lazily so simply default-constructing an SslContext does not pull in mbedTLS state.

Thread Safety
Mixed. Configuration calls (setCertificate, setPrivateKey, setCaCertificates, etc.) must be made before the context is handed to any SslSocket — the configuration phase is single-threaded. After that the context is read-only and may be shared across threads; mbedTLS's mbedtls_ssl_config is documented as safe to share across handshakes once initialized.
Example
SslContext ctx;
ctx.setProtocol(SslContext::SecureProtocols);
ctx.setCertificate(FilePath("/etc/server.crt"));
ctx.setPrivateKey (FilePath("/etc/server.key"));
httpServer.setSslContext(ctx);

◆ hasTlsSupport()

static bool promeki::SslContext::hasTlsSupport ( )
static

Reports whether this build can actually speak TLS.

Single source of truth for the PROMEKI_ENABLE_TLS feature flag: when false every SslContext mutator (other than the trivial protocol / verifyPeer / verifyDepth setters) returns Error::NotSupported and nativeConfig returns nullptr, so consumers can check this once instead of inspecting build macros.

HttpClient::hasTlsSupport, HttpServer::hasTlsSupport, and WebSocket::hasTlsSupport delegate here.

◆ setSystemCaCertificates()

Error promeki::SslContext::setSystemCaCertificates ( )

Loads the host platform's default CA bundle.

On Linux this probes the well-known locations (/etc/ssl/certs/ca-certificates.crt etc.) until one succeeds. Returns Error::NotExist if no bundle can be located.

◆ setVerifyPeer()

void promeki::SslContext::setVerifyPeer ( bool  enable)

Enables or disables peer-certificate verification.

Defaults to true. Disable only for development (e.g. self-signed loopback testing). When disabled a verification failure is tolerated but still reported via SslSocket::sslErrorsSignal.
