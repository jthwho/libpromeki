Detailed Description

template<typename T>
class CRC< T >

Table-driven cyclic redundancy check (CRC) computation.

Template Parameters
TUnsigned integer holding the CRC register and result. One of uint8_t, uint16_t, uint32_t, or uint64_t.

The class follows the Rocksoft / "CRC catalogue" parameter model (https://reveng.sourceforge.io/crc-catalogue/all.htm): every configurable CRC variant is described by a small Params struct holding poly, init, xorOut, and a single reflect flag that controls both input and output bit reflection (the asymmetric refIn != refOut combinations are exotic and not supported here).

Each CRC instance owns a 256-entry lookup table that is computed once in the constructor. After that, update is a tight byte-at-a-time loop with one table read, one XOR, and one shift per input byte — fast enough for video-rate use without any further vectorisation.

Example
CRC<uint8_t> crc(CrcParams::Crc8Autosar);
crc.update(payload, 8);
uint8_t value = crc.value();
// Or one-shot:
uint8_t value2 = CRC<uint8_t>::compute(CrcParams::Crc8Autosar, payload, 8);
// Or via the named factory:
Crc8 c = crc8_autosar();
c.update(payload, 8);

Verifying parameters against published catalogue check values
The Rocksoft catalogue lists a "check" value for each named CRC: the 8-bit ASCII string "123456789" (without the trailing nul) fed through the CRC must produce a known result. The unit tests for this class use those check values to validate the CrcParams presets.

One-shot CRC over a buffer.

Equivalent to constructing a fresh CRC, calling update once, and reading value. Convenient for callers that do not need to reuse the table. For frequently-computed CRCs that share parameters, prefer to keep a long-lived CRC instance and reset it between messages so the table is built only once.
