Detailed Description

Live wire from one MediaIOSource to one or more MediaIOSink.

A MediaIOPortConnection drives a signal-driven drain pump that forwards frames from a source port to one or more sink ports with proper backpressure. It subscribes to the source's MediaIOSource::frameReady and to each sink's MediaIOSink::frameWanted signal, and pumps the transfer until either the source signals end-of-stream / error, or every sink has stopped accepting frames (limit hit, write error, or removed).

Drain semantics
Reading from a MediaIOSource is destructive — only one consumer can pop a given frame off the source's queue. To support pipeline fan-out (one source feeding many sinks) the connection itself owns the read side and dispatches each frame to every active sink in turn.

On every frameReady or frameWanted, the connection drains non-blocking reads from the source into non-blocking writes on every non-stopped sink for as long as everyone cooperates:

  • readFrame returns Error::TryAgain → wait for the next frameReady.
  • readFrame returns Error::EndOfFile → emit upstreamDone and stop reading.
  • readFrame returns another error → emit errorOccurred and stop pumping.
  • Any active sink's writesAccepted drops to zero → wait for the next frameWanted before reading again (so the source's queue can absorb backlog while we wait, but the slowest sink throttles the whole connection).
  • A sink writeFrame returns a hard error → emit sinkError for that sink, mark it stopped, continue the pump for the remaining sinks.
  • A sink reaches its per-sink FrameCount limit at the next Frame::isSafeCutPoint → emit sinkLimitReached, mark the sink stopped (limit-reached), continue the pump for the remaining sinks.
  • All sinks become stopped → emit allSinksDone and stop pumping.
Frame-count cap
Each sink may carry an optional FrameCount limit (passed to addSink). An empty / unknown FrameCount means "no limit"; a finite, non-zero count caps the number of frames written to that sink. The cap is honoured at the next safe-cut-point on or after the cap-th write so the GOP / audio packet boundary containing the cap stays complete — the cut frame itself is dropped and the sink is marked stopped.
Lifecycle
Constructed in the stopped state. Sinks are added via addSink before start. Once start has been called, the sink set is frozen for the life of the run. stop disconnects the signal handlers and emits stopped; the connection does not open or close the underlying ports — the caller is responsible for opening / closing the owning MediaIO instances.
Threading
Slot delivery follows the connect-side EventLoop affinity, so the pump runs on whichever loop the MediaIOPortConnection lives on. Distinct connections may run on distinct loops.
1:1 convenience
The two-port (source, sink, parent) constructor remains as a thin convenience over the multi-sink form for the common standalone case (file-to-file copy, "save N frames" CLI helpers, test fixtures). It is equivalent to constructing a multi-sink connection with the given source and immediately calling addSink with no frame limit.
Use cases
Standalone tooling that needs source→sink forwarding without spinning up a full MediaPipeline; the pipeline itself uses one MediaIOPortConnection per producing stage to drive fan-out and per-sink frame-count caps.

◆ start()

Error promeki::MediaIOPortConnection::start ( )

Wires the drain handlers and primes the pump.

Connects the source's frameReady signal and each sink's frameWanted / writeError signal to the internal pump, then performs an initial drain attempt to pick up any frames already queued or kick a pre-fetch. Idempotent — calling start on a running connection is a no-op.

Returns
Error::Ok on success, or Error::Invalid when the source is null or no sinks are attached.

◆ stop()

void promeki::MediaIOPortConnection::stop ( )

Disconnects the drain handlers and emits stopped.

In-flight strand commands on the source / sinks complete normally — the connection just stops listening for frameReady / frameWanted / writeError. Idempotent.
