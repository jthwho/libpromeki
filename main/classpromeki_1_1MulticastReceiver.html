Detailed Description

Standalone multicast datagram receiver with an owned worker thread.

MulticastReceiver is a reusable primitive for any subsystem that needs to consume UDP datagrams from one or more multicast groups without having to build its own socket + thread plumbing. It owns a UdpSocket bound to a caller-chosen port, joins any number of Any-Source (ASM) or Source-Specific (SSM) groups on a single interface, spawns a dedicated receive Thread, and delivers every arriving datagram to a user callback.

Downstream consumers currently targeted:

  • MediaIO_Rtp reader mode (one receiver per video / audio / data RTP stream whose destination happens to be a multicast group).
  • SAP / SDP announcement listener (joins 224.2.127.254:9875 and hands each SAP PDU to the caller).
  • mDNS responder / resolver (joins 224.0.0.251:5353).
  • Any future AV-over-IP discovery / control protocol.
Threading model

The receive loop runs on a promeki Thread subclass named by the caller (via setThreadName, default "multicast-rx"). The thread does not use its built-in EventLoop — it blocks in UdpSocket::readDatagram with a short receive timeout so the stop flag can be polled between datagrams. The thread is joined automatically in stop() and in the destructor.

Delivery

The datagram callback is invoked from the receive thread. It receives:

  • a Buffer::Ptr whose backing allocation is owned by the receiver and valid only for the duration of the call — if the consumer needs to keep the bytes longer it must copy them or take a fresh Buffer::Ptr::create copy of the data.
  • the sender's SocketAddress.

Callbacks should be fast and non-blocking. Work that cannot complete inline (decoding, reassembly, upstream signalling) must be pushed onto a queue for another thread to consume.

Example — SAP listener skeleton
MulticastReceiver rx;
rx.setLocalAddress(SocketAddress::any(9875));
rx.setThreadName("sap-rx");
rx.setDatagramCallback([](Buffer::Ptr data, const SocketAddress &sender) {
// Parse the SAP PDU; hand the embedded SDP to the app.
});
Error err = rx.addGroup(SocketAddress(Ipv4Address(224, 2, 127, 254), 9875));
if(err.isError()) return err;
err = rx.start();
if(err.isError()) return err;
// ... later ...
rx.stop();

Lifetime

Configuration (local address, interface, groups, callback) must be set before start(). Once running, stop() is the only legal mutator; reconfiguration requires a stop / reconfigure / start cycle. The destructor calls stop automatically.

Thread Safety
Configuration setters and start / stop are intended to be called from the receiver's owning thread. The user callback is invoked on the dedicated receive thread spawned internally — the callback's body must be thread-safe with respect to whatever state it touches.

Sets the poll interval used to check the stop flag.

The underlying socket uses SO_RCVTIMEO set to this value so the receive loop wakes up periodically even when no packets are arriving. Lower values reduce stop-response latency at the cost of more syscalls on idle streams. Default is DefaultReceiveTimeoutMs.
