Detailed Description

Cross-process frame transport over shared memory + local sockets.

A FrameBridge publishes or consumes Frame objects across a process boundary on the same host. The output side creates a shared memory ring of frame slots and a control socket; input sides connect to the control socket by name, map the shm read-only, and read slots as new frames are announced.

Each input opts into one of two pacing modes at openInput time:

  • Synchronous (default): the input must acknowledge every TICK before the output's writeFrame returns. The output's cadence is bounded by the slowest sync input, which makes back-pressured pipelines trivial. Sync ACKs are emitted from inside readFrame after a fresh slot has been consumed.
  • Asynchronous (no-sync): the input never acknowledges and the output never waits for it. Readers that fall more than Config::ringDepth frames behind miss frames — signalled via framesMissed — and resynchronize on the next TICK.

Mixing modes is supported: an output may have both sync and no-sync inputs attached, and only the sync inputs gate the writer.

Waiting for consumers
With Config::waitForConsumer set, the output-side writeFrame blocks until at least one consumer is connected instead of silently discarding frames. This matches the common "producer stalls until downstream is ready" pattern without requiring callers to poll connectionCount themselves.
Wire protocol
The control channel uses KlvFrame framing. A HELO / ACPT handshake establishes wire-version compatibility and carries a config_hash that rejects readers whose expected shape does not match the publisher. Per-frame TICK messages announce new slot publications. See the class-level Doxygen on KlvFrame for details of the framing; the protocol identifiers are documented in the implementation.
Versioning
Each instance has a fresh per-create UUID, used for logs and cross-host identification. The shm header carries a wire-format version; major mismatches are rejected at handshake time, minor mismatches are accepted with a log entry.
MVP scope
The MVP supports one image plane group (one ImageDesc in the MediaDesc) and one audio track. Multi-image and multi-audio frames are planned work. Writes are always-copy — a future extension will add zero-copy via a dedicated MemSpace.
Cross-user access
Pass a non-default Config::accessMode (for example 0660) and a Config::groupName that both ends share. The underlying shm object and socket file are chmod'd and chown'd at openOutput time.

◆ abort()

void FrameBridge::abort ( )

Interrupts any in-flight blocking wait.

Thread-safe: may be called from any thread at any time. Sets an atomic flag that causes a writeFrame currently blocked inside a Config::waitForConsumer wait (or waiting on sync-input acknowledgements) to return Error::Cancelled promptly. Later writeFrame calls also return Error::Cancelled until the bridge is closed and reopened, so the caller sees a clean stop rather than silently resuming.

The typical use is a signal / quit handler that needs to break the producer out of its wait so the owning pipeline can close cleanly. See the MediaIO wrapper's cancelBlockingWork hook for the framework-level integration.

◆ lastFrameTimeStamp()

TimeStamp FrameBridge::lastFrameTimeStamp ( ) const

Returns the timestamp of the most recent frame.

On the output side this is the steady-clock time captured just before the frame was published to the ring (i.e. placed in the output queue). On the input side this is the publisher's timestamp carried by the most recently drained TICK. Returns a default-constructed (epoch) TimeStamp when no frame has been published/received yet.

Returns
The last frame's publish timestamp.

◆ service()

void FrameBridge::service ( )

Accepts pending inputs and prunes disconnected ones.

writeFrame already services accepts implicitly, so typical usage does not need this. Call it when a publisher needs to accept new inputs while not actively writing frames — for example while waiting for the first frame to arrive from upstream, or in tests.

No-op when not open as an output.

◆ writeFrame()

Error FrameBridge::writeFrame ( const Frame::Ptr &  frame)

Publishes a frame to all connected inputs.

Serializes the frame's images, audio, and metadata into the next ring slot, increments the slot sequence counter twice (seqlock), and sends a TICK to every connected input. A no-op (other than processing accepts and drops) when zero inputs are connected.

Parameters
frameThe frame to publish (must be compatible with the configured mediaDesc and audioDesc).
Returns
Error::Ok on success, Error::OutOfRange when audio or metadata exceeds slot capacity, or another error.
