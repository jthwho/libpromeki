Detailed Description

Abstract controller for media I/O backends.

MediaIO is the abstract root of the strategy hierarchy:

MediaIO (abstract — public API + cached state +
ports + signals; one virtual: submit())
└─ CommandMediaIO (executeCmd virtuals + dispatch helper)
├─ InlineMediaIO (submit runs inline on the calling thread)
├─ SharedThreadMediaIO (submit posts to a ThreadPool Strand)
└─ DedicatedThreadMediaIO (submit posts to an owned worker thread)

All operations — open, close, read, write, seek — are dispatched as MediaIOCommand instances through the single submit virtual. Each public method on MediaIO, MediaIOSource, MediaIOSink, and MediaIOPortGroup builds a concrete command, wraps it in a MediaIORequest, and calls submit to dispatch. Strategy subclasses choose the thread on which the command runs.

Backends register themselves via PROMEKI_REGISTER_MEDIAIO_FACTORY (declaring a concrete MediaIOFactory subclass) and are instantiated through the config-driven factory create or the convenience helpers createForFileRead / createForFileWrite.

Cached state

MediaIO owns the cached descriptor and navigation state populated from per-command Output fields. The cache is updated only by completeCommand (running on whatever thread the strategy chose), in this fixed order: cache update → signal emission → request resolution. .then() callbacks therefore always observe up-to-date cached state.

Non-blocking I/O

Per the always-async API rule every public method returns a MediaIORequest. Callers that want synchronous behavior append .wait(). MediaIOSink::writeFrame applies an always-on capacity gate that returns Error::TryAgain when the sink is full; consumers wait on frameWanted before retrying.

Thread Safety

Inherits ObjectBase — thread-affine. MediaIO is intended to be driven from a single user thread. Public methods (open / close / readFrame / writeFrame / seekToFrame / setStep etc.) are not safe to call concurrently from multiple threads. Cached state is updated inside completeCommand on the strategy's chosen execution thread; user-thread reads of cached accessors must observe the cache only after the relevant request resolves. Cross-thread notifications use signals (frameReady, frameWanted, writeError), which the signal/slot system marshals via the receiver's EventLoop.

Applies the standard Output* overrides from config to input, returning the resulting MediaDesc.

Every transform backend (CSC, FrameSync, SRC, VideoEncoder, VideoDecoder) consumes the same family of MediaConfig keys to express "given this input shape, produce this output":

  • MediaConfig::OutputPixelFormat — replaces the ImageDesc::pixelFormat of every video image.
  • MediaConfig::OutputFrameRate — replaces MediaDesc::frameRate.
  • MediaConfig::OutputAudioRate — replaces AudioDesc::sampleRate on every audio entry.
  • MediaConfig::OutputAudioChannels — replaces AudioDesc::channels on every audio entry.
  • MediaConfig::OutputAudioDataType — replaces AudioDesc::format on every audio entry.

Keys at their default (invalid PixelFormat, invalid FrameRate, zero audio rate / channels, invalid Enum) mean "inherit from input" — the corresponding field passes through unchanged.

The pipeline planner uses this helper to compute what shape a configured transform will produce without instantiating it. Backend authors call it from proposeOutput / describe so every transform answers the planner uniformly.

Parameters
inputThe MediaDesc the transform will consume.
configThe transform's MediaConfig (read-only).
Returns
The MediaDesc the transform will produce.

◆ applyQueryToConfig()

static Error promeki::MediaIO::applyQueryToConfig ( const Url &  url,
const Config::SpecMap &  specs,
Config *  outConfig 
)
static

Applies URL query parameters to a config via the backend's spec map.

Every key in url.query() is looked up in specs by its declared name (the identifier passed to PROMEKI_DECLARE_ID). The associated raw string value is coerced to the spec's expected Variant type via VariantSpec::parseString and range-checked via VariantSpec::validate before being written to outConfig. Values flow through the same coercion path as JSON/CLI entry, so a URL, a JSON config file, and a command-line flag all have identical type rules.

This is automatically invoked by createFromUrl after the backend's own MediaIOFactory::urlToConfig has handled the authority / path components, so most backends never need to call it directly.

Case sensitivity
Query keys and values are matched case-sensitively. A URL like "pmfb://x?framebridgeringdepth=4" is rejected with Error::InvalidArgument; the canonical "FrameBridgeRingDepth" spelling is required. This matches the formal RFC 3986 position (query components are case-sensitive) and — more importantly — closes the door on a class of silent bugs where a near-match name could resolve to the wrong key (e.g. FrameBridgeSynch vs FrameBridgeSync). The spec ID spelling is canonical in log lines, JSON configs, and --help output, so requiring the same spelling in URLs keeps grep/search coherent across all four layers. Scheme and host remain case-insensitive per RFC 3986 and are handled by Url::fromString itself.
Failure cases
All failures are fatal — on the first bad key the function logs a single warning and returns the corresponding error without writing that key (or any subsequent keys) to outConfig:
  • Error::Invalid — outConfig is null.
  • Error::InvalidArgument — a query key is not declared in any PROMEKI_DECLARE_ID for this Config type (including case-variant spellings), or is declared globally but not present in specs (i.e. the backend does not honor it).
  • Error::ConversionFailed — VariantSpec::parseString could not coerce the raw string into the spec's expected type.
  • Error::OutOfRange — the parsed value lies outside the spec's declared [min, max].
Parameters
urlThe URL whose query map to apply.
specsThe spec map defining accepted keys.
outConfigConfig to populate (must be non-null).
Returns
Error::Ok on success, or one of the error codes listed above.

◆ close()

MediaIORequest promeki::MediaIO::close ( )

Closes the media resource.

Submits a CmdClose to the strand. When the close completes, completeCommand pushes a single synthetic EOS read result onto each source's queue (so signal-driven consumers receive exactly one trailing frameReady whose pop returns Error::EndOfFile), resets the cached descriptor state, and emits closedSignal with the close result.

Callers that want synchronous behavior write io->close().wait(). Callers that want to fire-and-forget can just discard the returned request; the close still runs to completion (the strand keeps the command alive).

While the close is in flight isClosing returns true and MediaIOSource::readFrame / MediaIOSink::writeFrame stop submitting new work (reads still drain any queued results so the consumer can observe the trailing EOS). Callers must not touch the cached descriptor accessors (mediaDesc, frameRate, etc.) between this call returning and closedSignal firing.

Returns
A request resolving with Error::Ok on success or Error::NotOpen if the MediaIO isn't open or is already closing.

◆ completeCommand()

void promeki::MediaIO::completeCommand ( MediaIOCommand::Ptr  cmd)
protected

Centralized cache-update + future-resolution path.

Called by the strategy after executeCmd returns (and after MediaIOCommand::result has been populated). Walks the command's typed Output fields, applies them to the cache, emits the appropriate cache-derived signals, and finally invokes the command's MediaIOCommand::markCompleted hook so the MediaIORequest fulfills its promise.

Order is fixed: cache update → signal emission → request resolution. .then() callbacks always observe up-to-date cached state.

Backends never call this — only strategies do.

◆ describe()

virtual Error promeki::MediaIO::describe ( MediaIODescription *  out) const

Populates out with a snapshot of this MediaIO's identity, role, and format landscape.

The wrapper fills in:

  • Backend identity (name, description) from the registered MediaIOFactory.
  • Role flags (canBeSource / canBeSink / canBeTransform) from the same MediaIOFactory.
  • Instance identity (name) from this MediaIO.
  • Cached canSeek, frameCount, frameRate, containerMetadata if open.

Then dispatches to the backend's describe to add format-specific fields (producibleFormats, acceptableFormats, preferredFormat) and any pre-open probe results.

Synchronous; safe to call any time (pre-open, while open, after close). Cheap when open (cached state is reused) and as cheap as the backend's probe implementation otherwise.

Returns

◆ open()

MediaIORequest promeki::MediaIO::open ( )

Opens the media resource.

Builds an MediaIOCommandOpen, dispatches it via submit, and returns a request the caller can .wait() on or attach a .then() continuation to.

The backend's executeCmd declares the ports it exposes via CommandMediaIO::addPortGroup / addSource / addSink — the resulting MediaIO is a source if any sources were added, a sink if any sinks were added, or both (a transform) if both kinds were. When the open succeeds, completeCommand populates the cached mediaDesc / audioDesc / metadata / frameRate from the primary port; on failure it auto-dispatches a cleanup Close on the same backend instance.

Per the always-async API rule, callers that want synchronous behavior write io->open().wait().

Returns
A request resolving with Error::Ok on success, Error::AlreadyOpen if the MediaIO is already open, Error::Invalid if no backend is attached, or whatever the backend's executeCmd returns.

Sends a backend-specific parameterized command.

Parameterized commands let backends expose operations beyond the standard open/close/read/write/seek set — for example, setting device gain, querying device temperature, or retrieving codec parameters. The meaning of name and the contents of params and the resolved MediaIOParams output are entirely backend-defined.

◆ stats()

MediaIORequest promeki::MediaIO::stats ( )

Queries the cumulative-aggregate runtime statistics.

Builds a MediaIOCommandStats command, dispatches it via submit, and returns the request. The resulting MediaIORequest::stats() (after .wait()) is the populated MediaIOStats — backend-specific cumulative keys filled in by CommandMediaIO::executeCmd(MediaIOCommandStats &) and the framework-managed standard keys (rate trackers, drop / repeat / late counters, strand backlog) overlaid by completeCommand. Marked urgent so telemetry pollers do not block behind a deep queue of real I/O.

The same MediaIOStats type appears on every command's MediaIOCommand::stats — for standard commands it carries per-command telemetry instead. See MediaIORequest::stats() for the unified accessor.

Returns
A request that resolves with Error::Ok on success or Error::NotOpen when the MediaIO is not open or is closing.

◆ submit()

virtual void promeki::MediaIO::submit ( MediaIOCommand::Ptr  cmd)
protectedpure virtual

The single virtual gate every command flows through.

Public API methods on MediaIO, MediaIOSource, MediaIOSink, and MediaIOPortGroup build a concrete command, wrap it in a MediaIORequest, and call this virtual to dispatch it. Strategy subclasses (InlineMediaIO, SharedThreadMediaIO, DedicatedThreadMediaIO) implement it; MediaIO itself is abstract.

Strategies are responsible for invoking CommandMediaIO::dispatch to drive the backend's executeCmd, recording the result in MediaIOCommand::result, and finally calling completeCommand to apply cache writes, emit cache-derived signals, and resolve the request.

Backends never see this virtual.

Implemented in promeki::DedicatedThreadMediaIO, promeki::InlineMediaIO, and promeki::SharedThreadMediaIO.

Signal< Error > promeki::MediaIO::closedSignal

Emitted when a close request completes.

Signal:

Fires after the backend's Close command has run, the synthetic trailing EOS has been pushed to the read queue, and the cached state has been reset. Carries the close completion Error. Fires for both blocking and non-blocking closes — blocking callers simply see it before close returns.

◆ commandCompletedSignal

Signal< MediaIOCommand::Ptr > promeki::MediaIO::commandCompletedSignal

Emitted after every command resolves through completeCommand.

Signal:

Carries the just-completed MediaIOCommand::Ptr — its result, stats, and any typed Output fields are fully populated by the time this fires. Fires on the strategy's chosen execution thread (strand / dedicated worker / inline), so subscribers must either be thread-safe or rely on signal/slot cross-thread marshalling via EventLoop.

Intended as the observability hook for pipeline-level telemetry collectors (MediaIOStatsCollector subscribes per stage to roll per-command timings into a windowed ring) — anyone needing to observe every command without subclassing the backend can hang behaviour off this signal. Fires unconditionally regardless of result, including cancellation, so subscribers must inspect result themselves when filtering out failures matters.
