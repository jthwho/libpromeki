Detailed Description

Controller for media I/O backends.

MediaIO drives a MediaIOTask backend through a serialized command queue running on a shared ThreadPool. All operations — open, close, read, write, seek — are dispatched as MediaIOCommand instances. The task implements only the executeCmd virtuals; MediaIO handles threading, signals, and result caching.

Backends register themselves via PROMEKI_REGISTER_MEDIAIO and are instantiated through the config-driven factory create() or the convenience helpers createForFileRead() / createForFileWrite().

Cached state

MediaIO caches the descriptor and navigation state reported by the task (mediaDesc, audioDesc, metadata, frameRate, canSeek, frameCount, currentFrame). The cache is updated only on the user thread after a command's future resolves, eliminating the need for mutexes around task data.

Non-blocking I/O

readFrame() and writeFrame() take an optional block parameter (default true). When false, the command is enqueued and the call returns Error::TryAgain immediately. Use frameAvailable() to poll for ready frames and isIdle() to check whether the strand is busy.

Threading model

MediaIO is intended to be driven from a single user thread. Public methods (open/close/readFrame/writeFrame/seekToFrame/setStep etc.) are not safe to call concurrently from multiple threads. The cached state (mediaDesc, frameCount, currentFrame, ...) is read and written only from the user thread; backend execution happens on the strand worker thread. Cross-thread notifications use signals (frameAvailable, frameWanted, writeError), which the signal/slot system marshals via the receiver's EventLoop.

◆ adoptTask()

Error MediaIO::adoptTask ( MediaIOTask *  task)

Injects an externally constructed backend task.

Normally MediaIO obtains its task from the registered factory via create(). Some backends, however, need constructor arguments that can't be expressed in a Config — for example, the SDL player task needs raw pointers to an SDLVideoWidget and an SDLAudioOutput supplied by the application. For those cases the caller constructs the task directly (via a free factory function in the backend's library) and hands it to MediaIO via this method.

The MediaIO must not be open when this is called, and must not already own a task. Ownership of task transfers to the MediaIO, which will delete it in the destructor. Passing nullptr is an error.

Parameters
taskThe pre-constructed backend task.
Returns
Error::Ok on success, Error::AlreadyOpen if the resource is open, or Error::Invalid if a task is already adopted or task is null.

◆ pool()

static ThreadPool & MediaIO::pool ( )
static

Returns the shared thread pool used for the worker.

All MediaIO instances share a single static ThreadPool. Each instance has its own Strand on top of the pool, so different MediaIOs run concurrently while their own commands stay serialized.

Sizing

The default thread count is std::thread::hardware_concurrency(). Backends that block in executeCmd (e.g. live capture devices waiting for the next frame) hold a pool thread for the duration of the read. If you have many such backends active simultaneously, the pool can starve. Resize before opening the first MediaIO via:

MediaIO::pool().setThreadCount(N);

◆ PROMEKI_SIGNAL() [3/5]

MediaIO::PROMEKI_SIGNAL ( frameReady  )

Emitted whenever a read command completes.

Signal:

Fires after every CmdRead completion — success, end-of-file, or error. Signal-driven consumers should respond by draining the ready queue with repeated readFrame(frame, false) calls until it returns Error::TryAgain (no more data available yet), Error::EndOfFile (stream exhausted), or another error. The signal intentionally does not distinguish success from failure: consumers always discover the outcome by reading from the queue.

◆ readFrame()

Error MediaIO::readFrame ( Frame::Ptr &  frame,
bool  block = true 
)

Reads the next synchronized frame.

Submits a CmdRead with the current step, then either blocks waiting for the result or returns Error::TryAgain immediately.

EOF semantics

Once a read returns Error::EndOfFile, MediaIO marks the stream as exhausted and stops issuing further prefetch reads. Subsequent readFrame() calls keep returning Error::EndOfFile (without going down to the backend) until the user calls seekToFrame or close to reset the state. This avoids spamming the backend with reads it has already said are done.

Parameters
frameReceives the frame on success.
blockIf true (default), blocks until ready.
Returns
Error::Ok, Error::TryAgain, Error::EndOfFile, or another error.

◆ setBenchmarkIsSink()

Marks this MediaIO as a sink for benchmark submission.

Sinks submit each frame's Benchmark to their attached reporter after the taskEnd stamp. Non-sink stages still stamp — so a pipeline's terminal stage sees all the accumulated timestamps from upstream — but do not submit, avoiding double counting.

◆ setPrefetchDepth()

void MediaIO::setPrefetchDepth ( int  n)

Sets the number of read commands to keep in flight.

MediaIO will top up the strand queue to this many outstanding CmdReads in readFrame(). Larger values give the user thread more headroom for high-throughput sources (e.g. capture devices); a value of 1 is fine for files and lightweight sources.

Calling this marks the depth as user-set; subsequent open() calls will not overwrite it with the task's default until close() resets the override.

Parameters
nNew depth, clamped to ≥ 1.

◆ setStep()

void MediaIO::setStep ( int  val)

Sets the step increment.

The step value is copied into each CmdRead so the task can advance position accordingly. Common values:

  • 1: normal forward (default)
  • -1: reverse
  • 2: 2x fast-forward
  • 0: hold (re-read same frame)

If the new step differs from the old, any prefetched read commands (which were submitted with the old step) are cancelled and any pending read results are discarded. Subsequent reads will use the new step.

Parameters
valThe new step value.

◆ validateConfigKeys()

static Error MediaIO::validateConfigKeys ( const String &  typeName,
const Config &  cfg,
ConfigValidation  mode,
const String &  contextLabel = String() 
)
static

Validates a config's key set against the registered specs.

Calls unknownConfigKeys and then applies the error policy selected by mode:

  • Lenient — emit a warning for each unknown key and return Error::Ok. Used when typos should be visible but not fatal (e.g. an interactive REPL or a long-running service that accepts loose config).
  • Strict — emit a warning for each unknown key and return Error::InvalidArgument. Used when callers want a typo to fail the open immediately (e.g. a CLI tool where a misspelled key almost certainly means the user got something wrong).

Warnings are emitted via promekiWarn and include the optional contextLabel so callers can make each line self-identifying (e.g. "mediaplay: input[TPG]") — the framework never embeds caller-specific strings on its own.

Parameters
typeNameThe registered backend name.
cfgThe configuration to check.
modeThe error policy to apply.
contextLabelOptional prefix for log messages.
Returns
Error::Ok or Error::InvalidArgument per mode. Always Error::Ok when there are no unknown keys.
