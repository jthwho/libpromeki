Detailed Description

Async HTTP/1.x client built on top of TcpSocket and llhttp.

The client mirrors HttpServer's threading and EventLoop story: the calling thread's loop (or Application::mainEventLoop when none is current) drives all socket I/O. Each call to send returns immediately with a Future ; the response (or any error) is delivered when the EventLoop has had a chance to drive the request to completion.

Thread Safety
Inherits ObjectBase — thread-affine. All state mutates on the client's owning EventLoop thread. Callers that block-wait on the Future must be running on a different thread than the one driving the EventLoop, otherwise the loop will never get a chance to advance the request — the canonical pattern is "client lives on the main loop, worker threads issue requests and block on their Futures".
Lifetime model
The client owns no persistent connection state — every request opens a fresh TcpSocket, sends one request, reads the response, and closes. Connection pooling and HTTP/1.1 keep-alive reuse are deliberate non-goals for v1; the surface stays small and predictable, and a follow-up can layer pooling on without changing the public API.
TLS
https:// URLs are rejected until SslSocket is wired in; the client returns Error::NotImplemented from send when the request URL's scheme is https.
Example
HttpClient client;
Future<HttpResponse> fut = client.get("http://localhost:8080/api/status");
auto [response, err] = fut.result(2000); // 2-second timeout
if(err.isOk() && response.isSuccess()) {
promekiInfo("got %d bytes", (int)response.body().size());
}

◆ setSslContext()

void promeki::HttpClient::setSslContext ( SslContext::Ptr  ctx)
inline

Attaches the SslContext used for https:// requests.

Without a context the client falls back to the default-constructed SslContext (TLS 1.2 / 1.3, peer verification on, no CA bundle) which will fail verification against any real server — set a proper context (with SslContext::setSystemCaCertificates for the system trust store, for example) before making https:// requests.

Always available regardless of PROMEKI_ENABLE_TLS; in a TLS-disabled build the supplied context is stored but its operations are inert (see SslContext) and an actual https request is later rejected with Error::NotSupported.
