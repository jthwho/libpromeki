Detailed Description

TcpSocket extension that wraps the byte stream in TLS.

SslSocket inherits TcpSocket so existing event-loop machinery (poll, addIoSource, IODevice read/write) works unchanged — the only differences for callers are:

  • Before any I/O, attach an SslContext via setSslContext.
  • After the underlying TCP connection is up, call startEncryption (client side) or startServerEncryption (server side) to drive the TLS handshake. Both calls are non-blocking; the handshake completes as the loop's I/O readiness events arrive.
  • encryptedSignal fires once the handshake succeeds.
  • read / write proxy through mbedtls_ssl_read / mbedtls_ssl_write transparently.
Thread Safety
Inherits TcpSocket / IODevice — thread-affine. An SslSocket binds to one EventLoop's thread for the duration of its life. The attached SslContext may be shared across many SslSocket instances on many threads — its mutable lifecycle ends before the first handshake.
TLS framing vs TCP byte stream
bytesAvailable reports the number of plaintext bytes that mbedTLS has already decoded and buffered, not the raw TCP receive-buffer size. This means a poll-then-read loop that checks bytesAvailable() will always see zero until enough ciphertext has arrived to produce at least one record.

◆ continueHandshake()

Error promeki::SslSocket::continueHandshake ( )

Continues an in-flight handshake one step.

Returns Error::Ok once the handshake has fully completed (and isEncrypted is true); Error::TryAgain when more I/O is needed (caller should re-invoke after the next ready event); a real error code on hard failure.

Used by event-driven consumers (such as HttpConnection) to drive the handshake to completion across multiple I/O wakes after the initial startEncryption / startServerEncryption call returned Error::TryAgain.

◆ startEncryption()

Error promeki::SslSocket::startEncryption ( const String &  hostname = String())

Initiates a client-side TLS handshake.

The underlying TCP connection must already be established (call connectToHost first). The handshake runs asynchronously; encryptedSignal fires on success, errorOccurredSignal fires on failure.

Parameters

◆ sslErrorsSignal

Signal< StringList > promeki::SslSocket::sslErrorsSignal

Emitted with a list of human-readable verification errors.

Fires once after the handshake when peer verification was attempted but produced one or more errors (e.g. self-signed cert, expired cert). When SslContext::verifyPeer is true the handshake still aborts; with verification disabled this is the channel for inspecting what would have failed.
