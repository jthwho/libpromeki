Detailed Description

Per-thread event loop providing event dispatch, timers, and posted callables.

EventLoop is not an ObjectBase — it is infrastructure that ObjectBase instances attach to. Each thread may have at most one active EventLoop, tracked by a thread_local pointer accessible via current().

Two execution models are supported:

  • exec() — blocks, processes events until quit() is called.
  • processEvents() — processes pending events and returns, suitable for WebAssembly or other environments where the host owns the run loop.

exec() is simply a loop calling processEvents(WaitForMore).

Example
EventLoop loop;
// Post work to this thread's event loop
loop.postCallable([]() {
promekiInfo("Running on event loop thread");
});
// Start a repeating 100ms timer
int timerId = loop.startTimer(100, []() {
promekiInfo("Timer fired");
});
// Run the event loop (blocks until quit)
int exitCode = loop.exec();

◆ addIoSource()

int EventLoop::addIoSource ( int  fd,
uint32_t  events,
IoCallback  cb 
)

Registers a file descriptor to be monitored by this EventLoop.

While registered, cb is invoked on the EventLoop's thread each time fd becomes ready for any of the requested events. The callback runs synchronously between processEvents iterations (not from inside poll), so it may call any EventLoop API including removeIoSource on itself.

Thread-safe: may be called from any thread. When called cross-thread, the EventLoop is woken so its next poll includes the newly registered fd.

The caller retains ownership of fd — the EventLoop never closes it. Close the fd only after removeIoSource has been called and the handle is no longer in use.

The API intentionally uses an opaque integer handle rather than an ObjectBase-receiver pattern: TUI and SDL register a handful of long-lived sources at startup and unregister them at shutdown, which maps naturally onto a return-the-handle-and-hold-it model. An ObjectBase-receiver variant can be layered on top later if a widget-like consumer emerges — don't add one speculatively.

Parameters
fdThe file descriptor to monitor.
eventsBitmask of IoEvent values requesting read / write readiness notifications.
cbCallback invoked with (fd, readyEvents) on each readiness event.
Returns
A handle >= 0 on success, or -1 on failure (invalid fd, empty cb, empty events, or unsupported on this platform).
Note
On non-POSIX platforms (Windows, Emscripten) this call currently returns -1 and records Error::NotImplemented in the library log. Emscripten cooperative loops should move to explicit non-blocking processEvents calls.

◆ stopTimer()

void EventLoop::stopTimer ( int  timerId)

Stops and removes a timer.

Thread-safe: may be called from any thread. When called from a thread other than the one running exec(), a no-op wake marker is pushed so the worker loop notices the change. A stopTimer() call issued from another thread can race against a timer callback that is already executing — the callback will finish running before the stop takes effect, so any resources the callback depends on must outlive that small window.

Parameters
timerIdThe ID returned by startTimer().
