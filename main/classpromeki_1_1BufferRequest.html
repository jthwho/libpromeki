Detailed Description

Asynchronous handle for a Buffer mapping operation.

BufferRequest is the value-type façade returned by every async Buffer entry point — Buffer::mapAcquire, Buffer::mapRelease, Buffer::copyTo. It mirrors MediaIORequest exactly: the underlying BufferCommand carries the result, while the request adds a uniform wait / isReady / then / cancel interface.

Lifetime
BufferRequest is a value type holding a BufferCommand::Ptr (a shared pointer) plus a small Error sentinel for the resolved(Error) factory path. Copying the handle copies the shared pointer. The command is freed when every handle goes away; backends that have not yet processed the command keep their own reference until they do.
Inline vs async resolution
Backends that complete inline (e.g. HostBufferImpl, where mapAcquire(Host) is a no-op) construct the request via resolved(BufferCommand::Ptr) — the request reports isReady true immediately and then fires the callback synchronously. Async backends post the command to a worker / strand and return the in-flight handle; the framework's completion machinery (on BufferCommand) handles the wake-up.
Callback marshalling
then captures the calling thread's EventLoop at the time of registration and dispatches the callback through that loop's postCallable when the request resolves. When then is called on a thread without an active EventLoop, the callback runs synchronously on the thread that resolves the request.

Returns the underlying command typed as CmdT.

Returns a raw pointer because BufferCommand uses the SharedPtrProxy path, so a shared dynamic-cast across the hierarchy is not available. The returned pointer's lifetime is tied to this BufferRequest handle (or another copy of it).

Unchecked static_cast — callers built the command knowing its concrete type, so a runtime mismatch is a programmer error. Returns nullptr when the request has no command (the resolved(Error) sentinel path).

static BufferRequest promeki::BufferRequest::resolved ( BufferCommand::Ptr  cmd)
inlinestatic

Builds a request that is already resolved with the result baked into cmd.

Used for short-circuit returns where the backend can fully service the request inline and wants to surface a typed output (e.g. host-mapped buffer's BufferMapCommand::hostPtr). The caller pre-populates the command's BufferCommand::result (and any output fields), then this factory marks the command completed so subsequent wait / then calls fire immediately.

◆ then()

Attaches a continuation.

Marshalled through the EventLoop active on the thread that calls then. When the request is already resolved, the callback fires immediately (still marshalled through the captured loop, if any). Only one callback may be registered per request — a second then replaces the first.

◆ wait()

Error promeki::BufferRequest::wait ( unsigned int  timeoutMs = 0)
inline

Blocks until completion or timeoutMs elapses.

Returns Error::Invalid when the request is empty (default-constructed), Error::Timeout when the deadline expired before the command resolved, or the underlying command's BufferCommand::result on completion. When the request was built from resolved(Error), err is returned immediately regardless of timeoutMs.

Parameters
timeoutMsMaximum time to wait in milliseconds. A value of 0 (the default) waits indefinitely.
