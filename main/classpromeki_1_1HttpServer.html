Detailed Description

Top-level HTTP/1.x server bound to a TCP port.

HttpServer is the public face of the HTTP stack: it owns a TcpServer that listens for connections, an HttpRouter that maps patterns to handlers, and the live HttpConnection instances that flow through it. Everything happens on a single EventLoop — the one that's current at construction time, or Application::mainEventLoop when none is current — so handlers run on a known thread without needing their own synchronization.

The model intentionally mirrors Go's net/http: register routes (or middleware) on the server, then call listen. Handlers receive a request and write into a response object that the connection serializes to the wire when the handler returns.

Thread Safety
Inherits ObjectBase — thread-affine. Construction captures EventLoop::current as the owning loop. If the server is constructed inside a worker Thread, the server uses that thread's loop; otherwise it falls back to Application::mainEventLoop. Cross-thread handlers should accept the request, snapshot what they need, do their work on a worker, and call HttpConnection::postResponse from the owning loop (typically via EventLoop::postCallable). Signal connections to slots on other threads are dispatched automatically by the existing signal/slot machinery.
Example
Application app(argc, argv);
HttpServer server;
server.route("/api/health", HttpMethod::Get,
[](const auto &, auto &res) { res.setText("ok"); });
server.use([](const auto &req, auto &res, auto next) {
promekiInfo("%s %s", req.method().wireName().cstr(), req.path().cstr());
next();
});
server.listen(8080);
return app.exec();

◆ WebSocketHandler

using promeki::HttpServer::WebSocketHandler = std::function<void(WebSocket *socket, const HttpRequest &request)>

Callback invoked once a WebSocket upgrade succeeds.

Receives the freshly-constructed WebSocket (already attached to the upgraded socket and in the WebSocket::Connected state) along with the HttpRequest that triggered the upgrade — useful for reading query parameters or path placeholders negotiated during routing. The request is a snapshot taken at handshake time, so its body and headers are stable for the duration of the call.

Lifetime of the WebSocket is owned by the application — the callback is responsible for re-parenting or storing it somewhere that outlives the connection.

Mounts a CRUD HTTP API over a VariantDatabase.

Adds five routes under mountPath:

  • GET mountPath — full snapshot as JSON object
  • GET mountPath/_schema — registered specs as JSON
  • GET mountPath/{key} — one value as JSON
  • PUT mountPath/{key} — accepts JSON, validates, stores
  • DELETE mountPath/{key} — clears the entry

When readOnly is true, the PUT / DELETE routes are skipped — useful for exposing a runtime configuration as a read-only introspection endpoint.

The database is captured by reference; its lifetime must outlive the server.

◆ routeWebSocket()

void promeki::HttpServer::routeWebSocket ( const String &  pattern,
WebSocketHandler  handler 
)

Registers a route that upgrades to WebSocket on pattern.

Validates the Upgrade / Connection / Sec-WebSocket-Key / Sec-WebSocket-Version headers, computes Sec-WebSocket-Accept, sends a 101 Switching Protocols response, and once that response has finished writing, hands the underlying socket off to a new WebSocket. The handler is invoked with the WebSocket on the server's EventLoop thread.

Subprotocol negotiation: when the client sends a Sec-WebSocket-Protocol header, the first listed subprotocol is echoed back verbatim. Applications needing finer control should fall back to route + a hand-written upgrade handler.

◆ setSslContext()

void promeki::HttpServer::setSslContext ( SslContext::Ptr  ctx)
inline

Attaches an SslContext for TLS termination.

When set, every accepted connection is wrapped in an SslSocket configured with ctx and the server-side TLS handshake is driven inside HttpConnection before any HTTP bytes flow. The supplied context must already carry a server certificate + private key.

Pass an empty pointer to disable TLS again.

Always available regardless of PROMEKI_ENABLE_TLS; in a TLS-disabled build the context is stored but its operations are inert (see SslContext) and the server simply does not start a TLS handshake.
