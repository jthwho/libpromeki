Detailed Description

Common base for a single I/O port hanging off a MediaIO.

A MediaIO is a multi-port container; the actual read/write surfaces are its MediaIOSource and MediaIOSink children (both deriving from MediaIOPort). A port carries:

  • an identity (a per-type PROfessional MEdia toolKIt and a human-readable name),
  • a back-pointer to its owning MediaIO (reachable through the port group),
  • mandatory membership in a MediaIOPortGroup, which holds the group's clock / step / position state and references every port that shares that timing.

Per-direction surface area (writeFrame / readFrame, format negotiation, signals) lives on the derived classes — see MediaIOSink and MediaIOSource. Per-port descriptor caching is added to this base incrementally as later refactor phases move it off MediaIO. Per-port accounting is intentionally absent: rate tracking, drop / repeat / late counts, and pending command counts all live on the MediaIOPortGroup, because a single backend tick advances the whole group at once.

Port groups
Every port belongs to exactly one MediaIOPortGroup. An "independent" port is simply a port that is the sole member of its group. A "synchronized" pair (e.g. paired audio + video on a single backend) shares one group, and the group enforces lockstep advancement. Ports do not hold their own clock or position state — those live on the group.
Lifetime
Ports are created by MediaIO during executeCmd(Open) and parented to their MediaIOPortGroup via ObjectBase parenting; the port group is in turn parented to the owning MediaIO. Destruction therefore cascades MediaIO → PortGroup → Port, so a port is automatically destroyed when its group is destroyed (or when the MediaIO is). Ports must not outlive their port group.
Thread Safety
Per-port accessors are read by user-thread callers and updated by the strand worker thread via signals. Direct reads of cached descriptor state are valid only when the port is open and the user thread is sequentially consistent with prior strand updates — see the per-direction documentation on MediaIOSource / MediaIOSink for details.

enum promeki::MediaIOPort::Role

Role of a port within its MediaIO.

The role is a structural property — sources read out of a backend, sinks write into one. The role is determined by which derived class is instantiated; MediaIOSource always has role Source, MediaIOSink always has role Sink.

Enumerator

Constructs a port and binds it to a port group.

The port is parented to group via ObjectBase parenting and is therefore auto-destroyed when the group (or its owning MediaIO) is destroyed. Callers (typically MediaIO helpers) are responsible for registering the port with the right per-type list on the owning MediaIO and for assigning the per-type PROfessional MEdia toolKIt.

Parameters
groupThe port group this port belongs to; must be non-null. The owning MediaIO is reached via group->mediaIO().
nameHuman-readable port name (e.g. "video", "audio.left"). May be empty; derived classes that want a type-and-index default name pass one they synthesize from index.
indexPer-type port index (sources and sinks have independent index spaces; src0 and sink0 may coexist).

Returns the MediaIOPortGroup this port belongs to.

Always non-null for a constructed port. The group holds the timing reference (Clock), step, current frame, and seek state shared by every port in the group. An "independent" port is just the sole member of a single-port group.
