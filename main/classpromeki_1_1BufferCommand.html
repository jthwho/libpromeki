Detailed Description

Base class for asynchronous Buffer operations.

BufferCommand mirrors the MediaIOCommand shape: each outstanding async operation on a Buffer (mapAcquire, mapRelease, copyTo) is represented by a concrete BufferCommand subclass that carries inputs (set by the public Buffer API) and outputs (set by the BufferImpl backend). A BufferRequest is the value-type façade callers see.

Lifecycle
  1. Built by a Buffer public API method. Inputs (target domain, flags) are populated; result is initialized to Error::Ok. A BufferRequest is constructed from the command's Ptr.
  2. Dispatched by the backend. HostBufferImpl completes inline — populates outputs and calls markCompleted before returning. Async backends (CUDA-device, FPGA, RDMA) post the command to a strand or worker thread and return the in-flight handle to the caller.
  3. Completed by the backend calling markCompleted. Wakes every blocked waiter and fires any registered continuation callback.

Commands are shared via SharedPtr<BufferCommand, false> with COW disabled — concrete commands are never cloned.

Thread Safety
The completion latch and callback slot are internally synchronized — markCompleted, waitForCompletion, and setCompletionCallback are safe to call from any thread. The Inputs / Outputs on concrete subclasses are written by exactly one producer (the backend) and read by exactly one consumer (the caller, after the request resolves); no synchronization is required for those fields beyond the happens-before that completion provides.

Member Data Documentation

◆ cancelled

Atomic<bool> promeki::BufferCommand::cancelled {false}

Cancellation flag — atomic for cross-thread polling.

Set by BufferRequest::cancel. Backends read this just before dispatching the command; if set, the backend should short-circuit with Error::Cancelled instead of performing the operation. Late cancellations (after dispatch has begun) are best-effort — backends that cannot interrupt their work let it finish.

◆ result

Error promeki::BufferCommand::result = Error::Ok

Outcome of the command.

The backend writes the final Error here before calling markCompleted. Defaults to Error::Ok so backends that complete inline only have to set the field on the failure path.
