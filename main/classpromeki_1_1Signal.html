Detailed Description

template<typename... Args>
class promeki::Signal< Args >

Type-safe signal/slot mechanism for decoupled event notification.

Allows connecting callable slots (lambdas or member functions) that are invoked when the signal is emitted. Template parameters define the argument types passed through the signal.

Thread Safety
Thread-safe. connect, disconnect, disconnectFromObject and emit may be invoked concurrently from any thread; an internal Mutex serializes mutations of the slot list, and emit takes a snapshot of the slot list under that lock then iterates the snapshot outside the lock. This means slots fired from a given emit reflect the connections that existed when the snapshot was taken — a slot connected from another thread mid-emit is not invoked by that emit, and a slot disconnected mid-emit is invoked exactly once if it was present in the snapshot. Reentrant connect / disconnect from inside a slot is safe because the iteration runs against the local snapshot, not the live list. Cross-thread delivery is supported by the connect(Function, ObjectBase *) overload, which routes invocations through the receiving ObjectBase's EventLoop via postCallable; that routing is thread-safe at delivery time.

template<typename... Args>
size_t promeki::Signal< Args >::connect ( Function  slot,
ObjectBase *  owner 
)

Connects a callable with an ObjectBase context (Qt-style cross-thread dispatch).

When the signal is emitted from a thread whose EventLoop differs from the owner's ObjectBase::eventLoop, the call is marshalled onto the owner's EventLoop via postCallable and invoked asynchronously there — matching Qt's Qt::AutoConnection semantics. When the signal emits on the owner's own thread the slot is invoked directly with zero overhead.

The overload exists as a peer to the untyped connect(Function, void*) so callers that pass an ObjectBase-derived this pointer get the correct threading behaviour without having to reach for ObjectBase::connect; plain void* callers keep the raw, same-thread-only semantics.

Args are captured by value into a std::tuple and unpacked after the post, so argument types must be copy-constructible.

Receiver-liveness guarantees
The bridge holds an ObjectBasePtr to owner (not a raw pointer) and registers an auto-disconnect cleanup on owner via ObjectBase::registerCleanup, so:
  1. Once ~ObjectBase starts on owner, no new slot invocations are queued — the slot is removed from this signal's slot list before destruction completes.
  2. Cross-thread callables that were already queued on owner's EventLoop before the destruction also short-circuit at dispatch time once owner's tracker map clears, so a callable that was posted just before the delete becomes a no-op rather than a use-after-free.

The framework therefore guarantees the user slot is not invoked once owner has finished destructing. That covers the common "lambda captures @c this" case. User slots that capture additional raw pointers must still ensure those remain valid for the slot's lifetime — capture an ObjectBasePtr (or any other tracker) and re-check inside the slot when in doubt.

template<typename... Args>
static VariantList promeki::Signal< Args >::pack ( Args...  args)
static

Packs the arguments into a VariantList.

Packs all the signal parameters into a VariantList object that can be used to marshal the data to either defer the emit or pass it as an event to another object (e.g. on another thread). Note, this will also make copies of any arguments passed by reference as the references probably won't be valid by the time you want to use the container.

The definition lives in signal.tpp so that signal.h does not pull the full variant.h through its 40+ member type headers. Callers that need to invoke pack() must include promeki/signal.tpp.

Parameters
argsThe signal arguments to pack.
Returns
A VariantList containing copies of all arguments.
