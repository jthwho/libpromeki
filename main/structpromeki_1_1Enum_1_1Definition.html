Detailed Description

Immutable per-type record stored in the global registry.

Definitions live for the lifetime of the process and are referenced by direct const pointer from Enum and Type — every accessor below is therefore lock-free.

Two storage models share the same struct layout so the rest of the class does not care which one it is looking at:

  • Static (the common case): declared via PROMEKI_REGISTER_ENUM_TYPE. Both the Definition and its Entry[] live in .data / .rodata — no heap allocation, no Map, no String copy. The caller keeps ownership of the memory.
  • Dynamic: allocated on the heap by Enum::registerType for truly dynamic cases (V4L2 menu controls, runtime-computed tables, …). Ownership transfers to the registry and is never freed.

The data members are public so the struct is a structural type, but treat them as read-only; the registry is the only writer, and it only writes typeIndex once at registration time.

◆ entryQualifiedLits

StringLiteralData* promeki::Enum::Definition::entryQualifiedLits = nullptr
mutable

Parallel array of fully-qualified literal-backed String caches — "TypeName::EntryName", one per entry, with entryCount elements, indexed in lock-step with entries and entryNameLits.

Populated by the registry at the same time as entryNameLits. Both parallel arrays share the same underlying "Type::Entry\0" buffer per entry — entryNameLits points at offset typeNameLen+2 into it — so the per-entry storage is one heap block.

Used by Enum::toString so the in-list case never has to concatenate; two Enums with the same value return Strings backed by the same immortal storage, so String equality short-circuits on pointer match.

◆ nameLit

StringLiteralData* promeki::Enum::Definition::nameLit = nullptr
mutable

Literal-backed String cache for the type name.

Populated by the registry at registration time. Accessors that return the type name as a String wrap this record via String::fromLiteralData so no bytes are copied — the same const char* that lives in rodata (static path) or in the registry's leaked heap block (dynamic path) is reused on every call.
