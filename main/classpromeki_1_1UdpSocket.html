Detailed Description

Datagram-oriented UDP socket with multicast support.

UdpSocket provides connectionless datagram communication over UDP. It supports unicast and multicast send/receive, including joining and leaving multicast groups with optional interface binding.

For connected-mode UDP (via connectToHost()), the standard IODevice read()/write() methods can be used. For connectionless operation, use writeDatagram()/readDatagram() with explicit destination/source addresses.

Thread Safety
Inherits IODevice — thread-affine. A single UdpSocket must only be used from the thread that created it.
Example
UdpSocket sock;
sock.open(IODevice::ReadWrite);
sock.bind(SocketAddress::any(5004));
// Send a datagram
SocketAddress dest(Ipv4Address(239, 0, 0, 1), 5004);
sock.writeDatagram("hello", 5, dest);
// Receive a datagram
char buf[1500];
SocketAddress sender;
int64_t n = sock.readDatagram(buf, sizeof(buf), &sender);

◆ setPacingRate()

Error promeki::UdpSocket::setPacingRate ( uint64_t  bytesPerSec)

Sets a per-socket transmit rate limit.

Uses SO_MAX_PACING_RATE, which is enforced by the fq qdisc on Linux (default on modern kernels). Packets submitted via writeDatagram() or writeDatagrams() beyond the configured rate are held by the qdisc and released on schedule, with zero per-packet CPU cost.

Pass PacingRateUnlimited to disable pacing.

◆ setTxTime()

Error promeki::UdpSocket::setTxTime ( bool  enable,
int  clockId = 11 
)

Enables per-packet transmit-time scheduling.

Sets SO_TXTIME on Linux, which lets individual datagrams carry a SCM_TXTIME nanosecond deadline via Datagram::txTimeNs. Requires the ETF qdisc; works best with NICs that support hardware TX scheduling (Intel i210, i225). Used by ST 2110-21 sender pacing and by precise AES67 pacing.

Disabling is a no-op — once the caller stops passing a non-zero Datagram::txTimeNs, subsequent sends behave like plain UDP without reconfiguring the socket.

Parameters
enableTrue to enable, false to disable.
clockIdClock ID for the deadlines (defaults to CLOCK_TAI to match PTP).
Returns
Error::Ok on success, Error::NotSupported if the platform does not implement it, or another error on failure.

◆ writeDatagrams()

int promeki::UdpSocket::writeDatagrams ( const DatagramList &  datagrams)

Sends a batch of datagrams in one syscall.

On Linux this uses sendmmsg(), which hands the kernel the entire batch in a single system call and avoids the per-packet context-switch cost of looping on sendto(). On platforms without sendmmsg() the implementation falls back to a sendto() loop with equivalent semantics.

Each datagram's data pointer must remain valid until this call returns. The kernel copies into its own buffers before this function returns; callers do not need to keep the data alive afterwards.

If txTimeNs is non-zero and transmit-time is enabled (see setTxTime()), the datagram carries a SCM_TXTIME cmsg and is held by the ETF qdisc until the target time.

On partial failure the kernel reports how many datagrams it accepted before erroring; this call returns that count. Callers that care about completeness should compare against datagrams.size().

Parameters
datagramsThe list of datagrams to send.
Returns
The number of datagrams accepted by the kernel, or -1 if none were sent.
