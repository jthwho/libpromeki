Detailed Description

Base object for promeki.

This object is used by promeki to provide certain objects with a base level of functionality which include:

  • Signals and slots
  • Some level of meta type and reflection

The object was modeled from the Qt QObject, although isn't quite as versatile but it trades off that versatility for not needing an external meta object compiler.

Thread Safety
Thread-affine: an ObjectBase instance belongs to the EventLoop captured at construction time and must only be used from that loop's thread. Use moveToThread to migrate ownership. Cross-thread interaction goes through Signal connections (which marshal arguments via VariantList and dispatch through the receiver's EventLoop) or via eventLoop()->postCallable. The ObjectBasePtr tracker is internally synchronized with a Mutex so it's safe to invalidate from a foreign thread when the tracked object is destroyed.

◆ deleteLater()

void promeki::ObjectBase::deleteLater ( )

Schedules delete this for the next iteration of this object's EventLoop.

Idiomatic Qt-style cross-thread teardown. Posts a callable to the object's eventLoop() that performs the actual delete, so the destructor always runs on the object's affinity thread regardless of where deleteLater is invoked from. Safe to call from any thread.

Typical use: a worker-thread object that needs to be torn down by the owning thread. Calling worker->deleteLater() then thread->quit() works correctly because EventLoop drains every remaining queued item — including any cleanup callables posted from inside the destructor — before exiting.

Detaches the object from its parent before posting the delete-callable. Ownership transfers from the parent-child chain to the queued callable the moment deleteLater is invoked, so a parent that dies between now and the dispatch cannot double-delete the child via destroyChildren.

Falls back to a synchronous delete this when the object has no associated EventLoop. Always run the call as the last action that touches the object on the calling thread; nothing further may dereference this once the post lands on the target loop.

◆ registerCleanup()

void promeki::ObjectBase::registerCleanup ( ObjectBase *  target,
CleanupHandler  fn 
)

Registers a function to run during this object's destruction.

Cleanup handlers fire from the destructor (after aboutToDestroySignal emits, after children are destroyed) and receive this as their argument. Each handler is gated by an ObjectBasePtr to target — if target has already been destroyed by the time the handler would run, the framework skips it. Pass a non-null target when the handler dereferences a foreign object that may outlive this in the other direction; pass nullptr to register a handler that always runs.

Used internally to wire automatic signal/slot disconnect on receiver destruction — see Signal::connect with an ObjectBase context.

Parameters
targetForeign object the handler depends on (may be nullptr).
fnCleanup handler — must be non-empty.

◆ setParent()

void promeki::ObjectBase::setParent ( ObjectBase *  p)

Sets the parent of this object.

If the object already has a parent, it will be removed as a child from the old parent and added as a child to the new one.

Thread affinity
setParent is thread-affine: this and p must share the same owning Thread (or both have no Thread set). Cross-thread parenting is rejected at runtime via PROMEKI_ASSERT. Use deleteLater for the common cross-thread teardown pattern; for migrations of an entire subtree, call moveToThread on the root.
