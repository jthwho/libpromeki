# DataStream / DataType Consolidation

**Phase:** 7 (cross-cutting refactor)
**Dependencies:** Phase 1 (containers), Phase 2 (DataStream v3 wire format frozen), DataType registry (shipped 2026-05-15)
**Standards:** All code must follow `CODING_STANDARDS.md`. Every class requires complete doctest unit tests. See `README.md` for full requirements.

`DataStream` currently fans out to every serializable type in the
library: ~80 forward declarations, ~17 includes, and 141 `operator<<`
/ `operator>>` overloads in `include/promeki/datastream.h`, plus a
parallel `enum DataStream::Type { TypeUUID = 0x14, TypeTimecode = 0x16,
… }` wire-ID registry. Each new serializable type touches at least
three files: its own header/.cpp, `datastream.h`, and
`datastream.cpp`. The 2026-05-15 DataType registry shipped half of
the infrastructure needed to invert this — `DataType::Ops::{readStream,
writeStream}` already type-erase the per-type serialization — but the
per-type operators remain hand-rolled on `DataStream` itself.

This plan inverts the relationship: **each type owns its own
serialization**, the DataStream header keeps only primitive operators
plus a single templated `operator<<` / `operator>>` pair, and the
existing `enum DataStream::Type` wire-ID registry collapses into the
per-type metadata generated by a new `PROMEKI_DATATYPE` macro.

## Design summary

### The macro (in the class body)

```
PROMEKI_DATATYPE(Type, ID, currentVersion)
```

Three args, always. Generates a nested `promekiDataType` struct
holding:

- `id` — `static constexpr DataType::ID` (the wire tag, replaces
  `enum DataStream::Type` entries)
- `version` — `static constexpr uint32_t` (current writer version)
- `name` — `static constexpr const char *` (stringized `#T`)
- `type` — `static inline const DataType` (self-registers at
  static-init time via `DataType::registerType<T>`)
- `_readerTable` — `static inline const Map<uint32_t, fn-ptr>` built
  at compile time from `make_integer_sequence<uint32_t, currentVersion>`

The struct is *purely* framework-generated. User-authored
serialization lives as ordinary class members.

### Per-type user-authored API

```
class Timecode {
public:
    PROMEKI_DATATYPE(Timecode, 0x16, 3)

    Error                   writeToStream(DataStream &s) const;
    template <uint32_t V>
    static Result<Timecode> readFromStream(DataStream &s);

    // ...regular Timecode interface, untouched...
};
```

The macro emits the primary `readFromStream<V>` template with a
fail-loud `static_assert` body. The user supplies explicit
specializations for each `V` in `1..currentVersion` at namespace
scope in the .cpp:

```
template <>
Result<Timecode> Timecode::readFromStream<3>(DataStream &s) { /* V3 */ }
template <>
Result<Timecode> Timecode::readFromStream<2>(DataStream &s) { /* V2 */ }
template <>
Result<Timecode> Timecode::readFromStream<1>(DataStream &s) { /* V1 */ }
```

### Contiguous-versions policy

If `currentVersion = N`, readers for every version in `1..N` must
exist. Skipping/retiring versions is not supported — the framework
table builder folds over `make_integer_sequence<uint32_t, N>` and
references `T::template readFromStream<V>(s)` for each V; a missing
specialization instantiates the primary template, firing the
`static_assert` with a message naming the type and the range.

If version retirement is ever needed, a `minReadableVersion` field
can be added later without disturbing this design.

### Framework dispatch (one pair, lives in `datastream.h`)

```
template <Detail::DataStreamSerializable T>
DataStream &operator<<(DataStream &s, const T &v);

template <Detail::DataStreamSerializable T>
DataStream &operator>>(DataStream &s, T &v);
```

The concept gates on the nested `promekiDataType` struct + a member
`writeToStream(DataStream &) const -> Error`. Per-version readers are
not gated by the concept — the compile-time table builder enforces
their existence with a better error message than a concept failure.

The write template emits `beginFrame(D::id, D::version)`, calls
`v.writeToStream(s)`, then `endFrame()`. The read template reads the
frame header (with version), looks up the matching reader in
`D::_readerTable`, and calls it; unknown versions produce
`DataStream::UnsupportedVersion`.

## Files affected (framework)

- `include/promeki/datatype.h` — add `PROMEKI_DATATYPE` macro;
  add `Detail::buildReaderTable<T>` (consteval, fold over
  `integer_sequence`); add `version` field to `DataType` record
- `include/promeki/datastream.h` — delete ~80 forward decls, ~13 of
  the ~17 includes, all 141 per-type operator decls, and the per-type
  `enum DataStream::Type` entries; add the `Detail::DataStreamSerializable`
  concept and the one templated `operator<<` / `operator>>` pair
- `src/core/datastream.cpp` — delete all per-type operator
  implementations and their `writeXxxData` / `readXxxData` helpers
  (the type owns these now); keep primitive operators (`int8..uint64`,
  `float`, `double`, `bool`, `String`, `Buffer`, raw escape hatch)
- `src/core/variant.cpp` — simplify Variant serialization: it
  already dispatches through `DataType::Ops`; remove any leftover
  per-`Type*`-tag switches now that all types self-register
- Golden wire-format tests under `tests/unit/datastream_golden_*`
  must remain byte-identical — the per-type ID values migrate
  one-to-one from `enum DataStream::Type` into the macro's `ID` arg

## Files affected (per type, repeated ~80x)

For each migrating type T:

1. Header (`include/promeki/T.h`):
   - Add `class DataStream;` forward decl (if not present)
   - Add `PROMEKI_DATATYPE(T, OldEnumValue, currentVersion)` invocation
     in the class body (current version = 1 for almost everything,
     matching today's `beginFrame(..., 1)` literal)
   - Declare `Error writeToStream(DataStream &) const` member
   - Declare `template <uint32_t V> static Result<T> readFromStream(DataStream &)`
     (the macro provides the primary; user adds specializations)
2. Source (`src/.../T.cpp`):
   - Remove the existing `PROMEKI_IMPLEMENT_DATATYPE` /
     `PROMEKI_IMPLEMENT_DATATYPE_ID` invocation (the macro's
     `promekiDataType::type` static now does the registration)
   - Add `writeToStream` body (typically migrated from the
     existing `writeXxxData` helper in `datastream.cpp`)
   - Add `readFromStream<currentVersion>` specialization (typically
     migrated from the existing `readXxxData` helper)
3. `datastream.h`: delete the forward decl, the `operator<<` /
   `operator>>` decls for T, the `writeXxxData` / `readXxxData`
   decls, and the `TypeT = 0x??` enum entry
4. `datastream.cpp`: delete the matching definitions

## Wire format compatibility

The on-disk DataStream v3 format is preserved:

- Each per-type ID migrates *unchanged* from `enum DataStream::Type`
  into the macro's `ID` arg (e.g. `TypeTimecode = 0x16` →
  `PROMEKI_DATATYPE(Timecode, 0x16, 1)`)
- Frame header layout (`[tag(uint16)][version(uint8)][size(uint32)][body]`)
  is unchanged; the macro's `uint32_t version` is downcast to
  `uint8_t` at the wire boundary as it is today
- Per-type body encoding is preserved by lifting the existing
  `writeXxxData` body verbatim into `writeToStream` and the existing
  `readXxxData` body verbatim into `readFromStream<currentVersion>`
- Golden tests (`tests/unit/datastream_golden_*`) are the canonical
  check; they must remain byte-identical through Phase 3

## Open questions

1. **Templated types (`List<T>`, `Map<K,V>`, `Point<T,N>`, etc.).**
   Each instantiation needs a distinct wire ID, but C++ templates
   can't naturally vary a `static constexpr` across instantiations
   without explicit specialization. Two options to evaluate during
   Phase 3:
   - Hash the inner type's `DataType::ID` into a derived ID at
     macro/template expansion time (deterministic across builds if
     all referenced types have pinned IDs)
   - Keep these as today's "one outer ID per container kind, inner
     element framed separately" — works as long as readers can
     recover element type from the inner frames, which is already how
     `List<T>` operates per `datastream.cpp`
2. **Variant payload serialization.** `Variant` dispatches through
   `DataType::Ops::writeStream` / `readStream`. After this refactor
   those slots are populated by lambdas that delegate to
   `writeToStream` / `_readerTable`. The Variant-side code should
   simplify, but the *wire format* of a `Variant<T>` is byte-identical
   to a direct `T` write, and the v3 golden tests cover this. Confirm
   during Phase 1 that the slot wiring works for `Variant`'s
   roundtrip suite.
3. **TextStream operators.** Phase 2K (in `streams.md`) lists ~20
   types needing free `operator<<(TextStream &, const T &)`
   overloads. Those are independent of this refactor (TextStream is
   not DataStream), but the per-type `writeToStream` /
   `readFromStream<V>` pair could provide a natural mirror —
   `toTextStream` / `fromTextStream` — if we want to standardize the
   pattern. Out of scope for this plan; flag for future.
4. **Static initialization order across libraries.** `promeki-tui`
   and `promeki-sdl` link against `promeki` and may register
   additional types. Self-registration via `promekiDataType::type`
   fires at the consumer .so's load time, after `promeki`'s
   initialization. Confirm the DataType registry tolerates late
   registration (it should, by Construct-on-First-Use).

## Design notes (retained for reference)

### Why a function template, not separate named methods

An earlier design used `readFromStreamV1`, `readFromStreamV2`, …
named methods + a variadic `PROMEKI_DATATYPE(T, id, currentVer, oldVer1,
oldVer2, …)` macro and a hand-rolled FOR_EACH. The function template
approach (`template <uint32_t V> static Result<T> readFromStream(...)`)
plus the contiguous-versions policy eliminates the FOR_EACH macro
entirely — the table is built by `make_integer_sequence<uint32_t,
currentVer>` and a fold expression. The cost is the
contiguous-versions constraint, which is the right policy default
for wire-format types anyway.

### Why a nested struct instead of flat class members

`Timecode::id` / `Timecode::name` / `Timecode::version` as flat
`static constexpr` members on the class would collide with existing
member names on many types (`Color::name`, `Track::id`, …). Nesting
them in `promekiDataType` namespaces them cleanly. The nested struct
also gets access to the enclosing class's private members for free,
which matters if a future `writeToStream` reaches into the type's
internals without going through the public API.

### Why `version` as a parameter to `readFromStream<V>`

The version baked into the wire frame is the version the *blob* was
written with — not the type's current `T::promekiDataType::version`.
Encoding V as a template parameter on the reader (rather than a
runtime arg) lets each specialization be a separate function the
compiler can inline and optimize independently, and removes any
runtime `switch` from the reader bodies.

### Why preserve per-type wire IDs verbatim

The DataStream v3 wire format is frozen and protected by golden
tests. Migrating `TypeTimecode = 0x16` directly into
`PROMEKI_DATATYPE(Timecode, 0x16, 1)` means every existing on-disk
file, network capture, and persisted state continues to read
identically. No format bump, no compat shim.
