# Library Audit — 2026-04-25

## Goal

Comprehensive audit of `include/promeki/` and `src/` (core, network, proav, music, tui, sdl, csc) for:

1. **Coding standards compliance** — every file reviewed against `CODING_STANDARDS.md`.
2. **API quality** — opportunities to improve ergonomics, efficiency, consistency, and extensibility.
3. **Bugs and correctness** — latent bugs, missing error paths, lifetime issues, thread-safety hazards.

Scope is strictly `include/promeki/` and `src/`. `tests/`, `apps/`, `thirdparty/`, `docs/`, `cmake/`, `devplan/` are out of scope.

The library has ~282 headers and ~286 source files. A single-pass review is not realistic, so this plan breaks the work into phases by module, with per-file checklists, and a single rolling **Findings Register** that the reviewer appends to as issues are discovered. Findings are triaged and fixed in dedicated follow-up phases — the audit itself does not change code (apart from trivial doc fixes called out below).

---

## Audit Dimensions (Checklist Applied to Every File)

The reviewer applies this checklist to each file. Each dimension maps to a `category` tag used in the Findings Register.

### A. Standards compliance (`std`)

- [ ] File header block: `@file`, `@copyright Howard Logic. All rights reserved.`, "project root folder" wording with trailing period, `#pragma once`.
- [ ] Includes order: C++ stdlib → third-party → `<promeki/...>`. Angle brackets only.
- [ ] Namespace wrapped with `PROMEKI_NAMESPACE_BEGIN/END`. No `using namespace` in headers.
- [ ] Class layout matches the prescribed order (types, statics, ctors, operators, getters, setters, others).
- [ ] Indentation: 8-space tabs in code; 8-space soft for access specifiers, 16-space for member declarations.
- [ ] Naming: PascalCase types, camelCase methods, `_member` privates, `is`/`has`/`set`/`to`/`from` prefixes used correctly.
- [ ] Getters drop `get` prefix. No `bool *ok` parameters — use `Error` / `Result<T>` / `Error *err`.
- [ ] No raw `new` / `delete` for ownership; smart pointers (`SharedPtr`, `UniquePtr`) used per the Heap Ownership policy.
- [ ] Type aliases for templates: `Ptr`, `UPtr`, `List`, `PtrList`, `ImplPtr` defined where applicable; no raw `SharedPtr<X>` / `UniquePtr<X>` at call sites.
- [ ] `std::` types in public API wrapped via `using` aliases or replaced with promeki wrappers (`String`, `List<T>`, `Map<K,V>`, `Mutex`, …).
- [ ] TypeRegistry types (`PixelFormat`, `PixelMemLayout`, `ColorModel`, `MemSpace`) passed as `const Wrapper &`, not `::ID`.
- [ ] Well-known enums live in `enums.h` and use `TypedEnum<Self>` + `PROMEKI_REGISTER_ENUM_TYPE`.
- [ ] Blocking calls accept `unsigned int timeoutMs = 0` and return `Error::Timeout` on expiry.
- [ ] Timecode-style `return;` on void exits.
- [ ] No double-underscore reserved identifiers.

### B. Documentation (`doc`)

- [ ] Class `@brief` present; longer description if non-trivial.
- [ ] Thread-safety statement present (one of the four standard wordings).
- [ ] Every public method has `@brief` (trivial getters/setters may omit, but document non-obvious semantics).
- [ ] `@param`, `@return`, `@tparam` present where applicable. `@param[out]` / `@param[in,out]` used correctly.
- [ ] Stream operator overloads present where appropriate (`operator<<`/`operator>>` for `TextStream`, `DataStream`).

### C. API ergonomics & extensibility (`api`)

- [ ] Class category (Simple / Shareable / Functional / Utility) is correct and consistent with the rest of the library.
- [ ] `Result<T>` used for factory/parse/return-with-error methods; `Error` used elsewhere; no `bool` returns for error reporting.
- [ ] Public API uses promeki wrappers where they exist (`String`, `List<T>`, `Buffer::Ptr`, etc.).
- [ ] No raw template spelling at call sites (e.g. `SharedPtr<Buffer>` instead of `Buffer::Ptr`).
- [ ] Public members exposed only via accessors; no leaking `std::` types in the public API.
- [ ] `noexcept`, `const`, `constexpr`, `[[nodiscard]]` used consistently and correctly.
- [ ] Implicit conversions and operator overloads complete enough that callers don't need awkward workarounds (per "Fix the library, don't work around it").
- [ ] Naming consistent with similar classes (e.g., all containers use `pushToBack`, `popFromBack`, `isEmpty`).

### D. Bugs and correctness (`bug`)

- [ ] No raw `new` / `delete` outside the documented exceptions.
- [ ] Every code path that can fail returns/sets an `Error`.
- [ ] Thread-safety claim in Doxygen matches actual implementation (mutexes, atomics, ordering).
- [ ] Move/copy semantics consistent with category: data objects copyable, utility classes non-copyable/non-movable, ObjectBase non-copyable.
- [ ] No use-after-free or dangling raw pointers in `cleanup()`/`close()` paths.
- [ ] `errno` captured via `Error::syserr()` immediately after the failing system call (before any other syscall clobbers it).
- [ ] No sleep-loops, polling, or arbitrary timeouts substituted for proper synchronization.
- [ ] Containers checked for empty/range before subscript access.
- [ ] PIMPL impls have out-of-line destructor in the .cpp.
- [ ] `static_cast` / `reinterpret_cast` / C-casts checked for safety.
- [ ] No off-by-one errors in buffer/range arithmetic.
- [ ] Endianness, alignment, and signed/unsigned conversion bugs flagged.

### E. Logging (`log`)

- [ ] `PROMEKI_DEBUG(ModuleName)` registered in every `.cpp` that calls `promekiDebug`.
- [ ] Log messages include context (path, error, args), instance ptr (`ClassName(%p):`), and identifying name where available.
- [ ] Right severity (debug / info / warn / err) used.
- [ ] No `printf` / `cerr` / `std::cout` for logging.

### F. Testing (`test`)

Reviewer notes when a public API has no test coverage — but **does not write tests** as part of the audit. Findings of category `test` are logged for the test-coverage-enforcer to address later. This dimension is the only one explicitly out-of-scope for the audit's main remediation phases.

## Working Procedure

For each phase:

1. Read each header in the phase. Apply the dimensions checklist (A–F).
2. Spot-read the matching `.cpp` for any non-trivial header (look for bugs, missing logging, error paths, raw `new`/`delete`).
3. Append findings to the **Findings Register** with: ID, file, line(s), category, severity, description, suggested fix.
4. Mark the file's checkbox in the per-phase checklist.
5. After all files in a phase are reviewed, write a short phase summary at the end of the phase block: counts by category, top-3 themes, biggest risks.
6. Do **not** fix anything during the audit pass except trivial typos, broken Doxygen tags, and obvious dead comments (these are noted in the register as `fixed-in-pass` so the diff stays small).

After Phase 27, fixes are scheduled into prioritized remediation phases (defined at the end of this document, after we know what we found).

### Severity scale

- `crit` — known or near-certain bug, security issue, or undefined behavior.
- `high` — likely bug, broken contract, missing critical error path, thread-safety hazard.
- `med` — coding-standard violation that affects readability/consistency, or missing/wrong documentation on a non-trivial API.
- `low` — minor style, doc nit, cleanup opportunity.
- `idea` — feature or ergonomics suggestion (no defect implied).

- **R1** — `crit` and `high` bug fixes. Land first, individually if needed.
