# SMPTE ST 2110 Conformance

**Phase:** 3C / 4 follow-up
**Dependencies:** Phase 3A (UdpSocket / UdpSocketTransport), Phase 3C
(RtpSession / RtpPayload / SdpSession / RtpMediaIO), Phase 4 (MediaIO
framework, ClockDomain, MediaTimeStamp)
**Library:** `promeki`

**Standards:** All code must follow `CODING_STANDARDS.md`. Every class
requires complete doctest unit tests. See `README.md` for full
requirements.

This document is the conformance plan for the SMPTE ST 2110 family of
standards — `-10` system timing, `-20` uncompressed video, `-21`
traffic shaping, `-30` PCM audio, `-31` AES3 transparent, and `-40`
ancillary data. Initial development of the RTP stack happened
under `avoverip.md`; that document covered the building blocks
(`RtpSession`, `RtpPayload`, `SdpSession`, `MulticastManager`). This
document picks up from that working RTP framework and tracks the
SMPTE-specific work needed to be **conformant** and **interoperable**
end-to-end with managed-IP plants.

The project goal is a complete ST 2110-10/-20/-30/-40 implementation
plus enough -21 / -22 / -31 / 2022-7 scaffolding to land later without
re-architecting. We are still in initial development, so backwards
compatibility is not a concern.

We do **not** intend to deliver narrow-timing senders (Type N / Type NL)
from a stock Linux kernel without dedicated hardware — that requires
either a DPDK-style userspace stack or a custom NIC with packet
scheduling. We do intend the framework to be sufficiently decoupled
that a future DPDK or hardware transport drops in without disturbing
the packetizers, the SDP layer, the MediaIO backend, or the rest of the
pipeline.

### Audit confidence

This plan was assembled from a first-pass audit. Coverage is uneven —
the **Current state** section and the per-phase checklists below are
grounded where I read the relevant standard cover-to-cover and the
relevant code in depth, and **extrapolated from the SMPTE
restatement** where I did not. Each phase below has a **Required
reading** callout listing the standards and source files that must be
read in full before the phase is executed; that reading is part of the
phase's work, not a prerequisite. The callouts are also the gate where
the phase's checklist must be revisited — anything I claimed missing
that turns out to already be implemented gets struck; anything I
claimed solid that turns out broken gets a new item.

Specifically, what I have **not** yet read:

- **AES67** (any edition). ST 2110-30 normatively delegates §7.1
  (payload) and §7.5 (sender timing) to AES67. Items I called missing
  in Phase E30 might be AES67-required behaviour that the existing
  code partially satisfies.
- **ST 2022-7** itself. Phase E2022 is sketched from the ST 2110-10
  §8.5 summary, not the redundancy standard's wire-level provisions.
- **ST 2059-2** (SMPTE PTP profile) and **IEEE 1588**. The PTP plan
  defers to a host-side `ptp4l`; the SMPTE profile constraints
  (message rates, BMCA settings, traceability flags) are not yet
  audited.
- **RFC 4175** (raw video RTP), **RFC 7273** (ts-refclk), **RFC 7104**
  (DUP), **RFC 3550** / **3551** / **8866** / **4570** / **5888** /
  **8285** in detail. The -10 / -20 restatements I read are normative
  for the SMPTE constraints _on top of_ the RFCs but do not replace
  the RFC's own normative provisions.
- **ST 2110-22** in full.
- Most of `src/proav/rtpmediaio.cpp` (~4000 LOC), `src/network/
  rtpsession.cpp`, `src/network/rtppayload.cpp` past the RawVideo
  packer, every per-stream packetizer / depacketizer / TX thread,
  `rtcppacket.cpp` / `rtcpscheduler.cpp`, the seq tracker, the
  reorder buffer, the aggregator, and the tests.

### Risks register

- **True Type N requires hardware we don't have.** Stock Linux kernel
  pacing tops out at Wide-class behaviour. Mitigation: default
  self-label as Type W when on `SO_MAX_PACING_RATE`; Type NL is
  conditional on `SO_TXTIME` + PHC-locked NIC + RT priority and must
  be bench-verified before we claim it; Type N stays out of scope
  pending DPDK or custom-NIC work.
- **PTP slave is a large undertaking.** Embedding a full IEEE 1588
  slave with BMCA is months of work. Mitigation: Phase D2 default is
  host-side `ptp4l` + PHC ioctls (read `/dev/ptp0`). The embedded-
  slave path is reserved as a far-future option.
- **No reference 2110 hardware on the bench.** We can validate against
  software counterparts (`ffmpeg -i input.sdp`, GStreamer
  `rtpvrawpay`/`rtpvrawdepay`, EBU LIST analyzer, NVIDIA Rivermax
  samples if available) and self-loopback, but interop with a
  proprietary sender (AJA Kona IP, Riedel, Embrionix, Nevion) is
  untested until we have one on hand. Flag any production deployment
  without prior lab time as untested.
- **AES3-3 procurement blocks Phase E31.** No timeline. The phase
  stays deferred and the rest of the plan is unaffected.
- **First-pass plan content.** Anything outside the per-phase
  Required-reading callout may be wrong. Mitigation: each phase
  starts with its reading and revises its own checklist before code
  lands. See "Audit confidence" above.
- **No SMPTE PICS conformance test suite.** We will not have an
  independent witness that confirms checklist completion. Mitigation:
  exhaustive doctest coverage of pack / unpack round-trip, exhaustive
  SDP round-trip tests, and ffmpeg / GStreamer / EBU LIST as
  external sanity witnesses.

## Current state (2026-05-21)

### Conformant or nearly so

- **RTP framing** — `RtpPacket` is a BufferView over the standard
  12-byte RFC 3550 header, V=2, supports CSRC/extension; matches
  ST 2110-10 §6.2's reference to RFC 3550.
- **Dynamic PT range** — defaults at 96; payload-specific overrides
  keep dynamic types in 96-127. Matches ST 2110-10 §6.2.
- **Big-endian on the wire** — RtpPacket setters / RFC 8331 payload /
  RFC 4175 payload / L16 / L24 all encode big-endian.
- **RTCP demux** — RTP/RTCP-mux on a single port (RFC 5761) via byte[1]
  in `[200..223]` range. SR + SDES + BYE compound emission. Receive
  thread parses SR and exposes via `RtpSession::receivedSr()`.
- **RtpStreamClock** — RFC 3550 §6.4.1 anchor pair → NTP wallclock
  mapping. Same arithmetic on writer (`setRtpAnchor`) and reader
  (`receivedSr`).
- **RFC 8331 / ST 2110-40** — `RtpPayloadAnc` already implements
  packAncFrame / unpackAncPackets with the 8-byte payload header
  (ESN/Length/ANC_Count/F), the 4-byte per-packet record, the
  ST 2110-40 §5.5 keep-alive (`ANC_Count=0` with M=1), and the §2.1
  invalid F-bit guard. SDP `TROFF` round-trip wired through `AncDesc`.
  90 kHz clock per §5.3.
- **MulticastManager / MulticastReceiver** — IGMPv3 ASM/SSM group join,
  outgoing interface select, TTL configuration.
- **PacketTransport abstraction** — `PacketTransport` exposes
  `Datagram::txTimeNs` and `setTxTime()`.  Phase D1 added
  `PacketScheduler` which is the per-session policy layer that turns
  those primitives into either burst, userspace cadence, kernel-fq
  rate cap, or per-packet `SCM_TXTIME` deadlines.  The factory
  routes by `RtpPacingMode`; a future `DpdkPacketScheduler` /
  `IntelI225PacketScheduler` drops in without disturbing the RTP
  framing, SDP layer, or MediaIO backends.
- **SDP basics** — `SdpSession` reads/writes RFC 4566 syntax, preserves
  attribute insertion order, parses `a=rtpmap` and `a=fmtp` into
  structured forms. Round-trip through `fromFile()` / `toFile()`.

### Partial — works for "good-enough" interop, not conformant

- **RFC 4175 raw video (`RtpPayloadRawVideo`)**
  - Emits **one** SRD header per packet; never sets the **C
    (Continuation) bit**. RFC 4175 / 2110-20 §6.3.2 (GPM) and §6.3.3
    (BPM) both rely on packing multiple SRDs to avoid sub-1000-octet
    packets.
  - Only 8-bit interleaved formats are wired (`YUV8_422_UYVY_*`,
    `RGB8`). 10/12/16-bit pgroups, 4:4:4, 4:2:0, ICtCp, CLYCbCr, XYZ,
    Key/Alpha signals — all missing.
  - F-bit in the per-SRD header is always 0 — no interlaced or PsF
    support.
  - Extended sequence number is local to the packetizer (`extSeq`
    starts at 0 each `pack()` call) instead of carrying the high 16
    bits of a per-stream 32-bit packet counter that wraps across
    frames. Reader currently ignores the field. ST 2110-20 §6.1.4
    requires the high 16 bits of the extended 32-bit sequence.
  - Pixel offset on receive is decoded but receive path doesn't
    enforce ascending order within a frame.

- **PCM audio (RtpMediaIO + RtpPayloadL16 / L24)** (updated 2026-05-21 by Phase E30)
  - Both L16 (`PCMI_S16BE`) and L24 (`PCMI_S24BE`) wire formats
    plumbed through `configureAudioStream` and selected via
    `MediaConfig::RtpAudioWireFormat` (default `Auto` resolves to
    L24 at 96 kHz or for 24-bit sources, L16 otherwise).
  - Sample rates 44.1 / 48 / 96 kHz all supported by the packet-
    sizing math; the MTU-clamp fallback walks the standard AES67
    intervals {4 ms → 1 ms → 333 µs → 250 µs → 125 µs} when the
    requested packet time can't fit.
  - `a=ptime:<ms>[.<decimal>]` emitted in SDP per AES67 §8.1 and
    parsed back on the reader side into
    `MediaConfig::AudioRtpPacketTimeUs`.  Receiver-side rtpmap
    encoding seeds `RtpAudioWireFormat` when the caller left it
    `Auto`.
  - `channel-order=SMPTE2110.(...)` per ST 2110-30 §6.2.2 emits +
    parses through `AudioChannelMap::toSt2110ChannelOrder` /
    `fromSt2110ChannelOrder` and rides on `AudioDesc::toSdp` /
    `fromSdp`.  Suppressed for 1/2/3-channel streams per RFC 3190
    §3.  `222` (22.2) parses (24× Unused) but never emits —
    deferred until SMPTE ST 2036-2 is procured.
  - Conformance Level labels (Level A / AX / B / BX / C / CX or
    `None` when outside Table 2) computed at configure time and
    surfaced via `MediaIOStats::StatsAudioConformanceLevel` +
    `StatsAudioWireFormat`.
  - `RtpAudioMaxUdp` hard-rejects values >1460 (ST 2110-30
    §6.2.1 Standard UDP Datagram Size Limit).
  - Open: full AES67 §7.5 sender-jitter audit beyond the current
    `StallReanchorMultiplier=16` heuristic; RTCP RR-driven
    conformance diagnostics; AES67 §10 SIP connection management
    (out of scope of ST 2110-30).

- **SDP — ST 2110-10 system timing** (updated 2026-05-20 by Phase E10)
  - `ts-refclk` PTP / localmac / traceable forms parse and emit via a
    dedicated `Rfc7273RefClk` value class.  `traceable` is gated on
    `MediaConfig::RtpPtpTraceable` until a PTP slave (Phase D2)
    can supply the lock-state signal automatically.
  - `MAXUDP`, `TSMODE=SAMP|NEW|PRES`, `TSDELAY=<usec>` round-trip
    through fmtp on every writer stream.
  - `source-filter: incl IN IP4 <group> <source>` (RFC 4570) emits on
    every multicast section when `MediaConfig::RtpSourceAddress` is
    set; receiver-side round-trip works via session-level vs.
    media-level `a=` routing in `SdpSession::fromString`.
  - `group:DUP <primary-mid> <secondary-mid>` (RFC 5888 + RFC 7104)
    emits at session level with per-section `a=mid:` tokens when a
    `Rtp{Video,Audio,Data}DestinationSecondary` is configured.  Dual-leg
    transport is Phase E2022; this lands the SDP-level signalling.
  - `mediaclk:direct=<offset>` still emitted unconditionally;
    `mediaclk:sender` is not yet wired (open Phase E10 follow-up).

- **RTP transport (UdpSocketTransport)** (updated 2026-05-21 by Phase D1 + D4)
  - `setPacingRate` (`SO_MAX_PACING_RATE`) is wired and routed
    through `KernelFqPacketScheduler`; this caps byte rate, not
    per-packet deadlines.
  - `setTxTime()` is honoured.  `TxTimePacketScheduler` calls
    `transport->setTxTime(true)` at configure time and per-datagram
    `txTimeNs` deadlines are wired all the way down to the kernel's
    `SCM_TXTIME` cmsg (audited in `UdpSocket::writeDatagrams`).
    Without ETF + a launch-time-capable NIC the kernel ignores the
    cmsg and behaviour matches `KernelFq`; the scheduler still
    emits the SDP `TSDELAY` estimate.
  - **IP DF (Don't Fragment) bit is wired.**  Toggleable via
    `UdpSocketTransport::setDontFragment(bool)`;
    `RtpMediaIO::openStream` opts in by default
    (`MediaConfig::RtpDontFragment` = true) per ST 2110-10 §6.3.

### Missing entirely

- **ST 2110-21 traffic-shaping models** — no VRX, no CMAX, no PRS
  scheduling, no Type N / NL / W classification, no TPR_j math, no
  TROFF, no TP fmtp. (Acknowledged: a kernel-pacing Linux sender will
  practically be Type W at best; Type N requires NIC pacing.)
- **ST 2110-31 AES3 transparent** — no `RtpPayloadAm824` class, no AES3
  block/frame framing, no `audio/AM824` SDP, no V/U/C/P preservation.
  Out of scope of `-30` (which is PCM-only).
- **ST 2110-22 compressed video** — JPEG XS exists at the RFC 9134
  layer (`RtpPayloadJpegXs`) but the -22 SDP signalling layer
  (`a=fmtp:... TP=2110TPN; CMAX=...`, the `transmission-mode`
  parameters) is not wired.
- **ST 2022-7 redundant streams** — SDP-level `group:DUP` /
  `a=mid:<token>` now emit + parse (Phase E10).  Still missing:
  dual-network sender/receiver pair, seamless protection switching.
  Tracked under Phase E2022.
- **PtpClock** — fully missing (open item in `avoverip.md`). Without a
  real PTP slave the writer can only stamp `ts-refclk:localmac` or a
  static `ts-refclk:ptp` declaration; the receiver cannot validate that
  it shares a clock with the sender.  Tracked under Phase D2.

## Decoupling work (prerequisite for narrow timing)

Before any -21 traffic-shaping work lands, the RTP send path needs an
explicit packet-scheduling seam so the future DPDK / NIC-pacing
backend drops in without disturbing the rest of the stack. The
PacketTransport abstraction already covers _where bytes go_; the
missing piece is _when each packet is allowed to leave_.

### Phase D1 — `PacketScheduler` abstraction

**Status:** landed 2026-05-21.  All 4 backends in place, RtpSession
routes every batch through its scheduler, per-stream TX threads no
longer carry inline pacing logic (video) or were already inline-pacing
only (audio — kept its own `Cadence` to avoid double-paving).  Audio
runs against a `BurstPacketScheduler` so the inline cadence stays
authoritative; video and data honour the user's `RtpPacingMode`.

**Required reading (do before implementing):**
- `src/network/rtpsession.cpp` — full file. Confirm how `sendPackets`
  drives the transport today; the scheduler seam slots in here.
- `src/proav/rtpmediaio.cpp` per-stream TX threads
  (`VideoTxThread::run`, `AudioTxThread::run`, `DataTxThread::run`).
  These are the current callers of `RtpSession::sendPackets` /
  `setPacingRate`; understand the pacing loop they already implement
  before replacing it.
- `src/network/rtptxthread.cpp` and `include/promeki/rtptxthread.h` —
  the base TX thread class.
- `src/network/udpsockettransport.cpp` — read the existing
  `setPacingRate` implementation and the `sendmmsg` path. Confirm
  where `Datagram::txTimeNs` is dropped today (it's plumbed but
  unused — verify exactly where it stops).
- `include/promeki/cadence.h` and any callers — the existing userspace
  pacing helper that the `Userspace` mode uses today.
- Linux `man 7 socket` / `man 8 tc-etf` for `SO_TXTIME` and the ETF
  qdisc semantics that will back the eventual `TxTime` scheduler.

**Implementation checklist:**
- [x] Abstract base class.  Owns a borrowed `PacketTransport *`
      (caller-owned) via `setTransport`.
- [x] `Error configure(const Spec &)` carries `frameInterval`,
      `bytesPerSec`, `packetsPerFrame`, ST 2110-21 `trOffsetUs` and
      `maxUdp`.  The VRX / CMAX terms remain Phase E21 work — the
      Spec is the right shape to grow into them.
- [x] `int enqueue(const PacketTransport::DatagramList &)` — the
      RtpSession builds the Datagram list (one frame's worth of
      packets the session already addressed and header-stamped) and
      hands it off.  Scheduler subclasses translate to per-packet
      pacing.  Per-packet `txTimeNs` is wired by
      `TxTimePacketScheduler` only.
- [x] `Error flushPending()` / `Error abort()` on the base — no-ops
      for the synchronous backends.  Sketched for the eventual
      `DpdkPacketScheduler`.
- [x] Scheduler never touches RTP header content.  Packetizer stamps
      marker + RTP-TS; session fills version/seq/SSRC/PT; scheduler
      only paces.
- [x] `RtpSession::sendPackets(RtpPacketBatch &)` is now a shim that
      fills the transport-side header fields, builds the Datagram
      list (zero-copy refs into the batch's `RtpPacket` buffers),
      then hands off to the scheduler.  The no-scheduler fallback
      path (tests / bring-up) still loops over the transport
      directly.
- [x] `RtpSession::setPacingRate` forwards to `scheduler->setRate`.
      Audio's behaviour is preserved because we install
      `BurstPacketScheduler` on audio sessions — see "Audio's
      cadence stays inline" below.
- [x] `RtpPacingMode` enum kept; `PacketScheduler::create(const Enum
      &mode, PacketTransport *)` is the factory.  `Auto` resolves to
      `KernelFq` on Linux and `Userspace` elsewhere, matching the
      prior `RtpMediaIO::configurePacingMode` default.
- [x] TSDELAY autocomputation — the SDP builder reads
      `scheduler()->predictedTxDelayUs()` when the user has not
      pinned an explicit `RtpVideoTsDelayUs` / `RtpAudioTsDelayUs` /
      `RtpDataTsDelayUs`.  `BurstPacketScheduler` returns 0 (no
      attribute emitted); `Cadence` / `KernelFq` / `TxTime` return a
      microsecond estimate based on `frameInterval` and
      `packetsPerFrame`.

**Audio's cadence stays inline (deliberate):**

`RtpAudioTxThread` keeps its own `Cadence` + `sleepUntil` loop because
the silence-fill path needs the cadence tick to fire even when the
upstream packetizer's queue is empty — the scheduler can't make that
call.  `RtpMediaIO` installs `BurstPacketScheduler` on every audio
session regardless of `RtpPacingMode` so the inline cadence isn't
double-paved.  A future "audio uses Streamwide-cadence scheduler"
refactor (where the audio TX hands silence + packets through the same
scheduler) is feasible but out of scope here.

**Why this matters for narrow timing**
- A future `DpdkPacketScheduler` uses the same `enqueue()` API and an
  associated `DpdkPacketTransport` to bypass the kernel entirely.
- A future hardware backend (`IntelI225PacketScheduler` or similar)
  computes TPR_j against a NIC PTP clock and uses TXTIME or LAUNCHTIME
  packet descriptors.
- Tests can stub the transport and assert per-packet egress timing
  against a virtual clock — that's exactly what
  `tests/unit/network/packetscheduler.cpp` does today.

### Phase D2 — PTP-traceable wallclock anchor

**Status:** ✅ Landed 2026-05-21 (host-`ptp4l` path).  Embedded
IEEE 1588 slave deferred indefinitely — the linuxptp `ptp4l`
+ `/dev/ptpN` PHC ioctl path is the conformance default.

**Decision: host-side `ptp4l`, not embedded slave.**  The plan's open
question (embed vs. consume a host slave) resolved in favour of
consuming the kernel's PHC subsystem via `clock_gettime(FD_TO_CLOCKID)`
+ `PTP_CLOCK_GETCAPS` + `PTP_SYS_OFFSET_PRECISE` ioctls — `linuxptp`'s
`ptp4l` (and optionally `phc2sys`) does the wire-protocol work and
disciplines `/dev/ptpN`; the library reads from that device, not from
the wire.  An embedded slave would have been months of work and
covered exactly one deployment shape (the one where the host doesn't
already run `ptp4l`).  Re-open if a fielded use case demands it.

**Required reading (done):**
- `~/docs/ieee/2008-07-24 IEEE 1588-2008 ...pdf` — message format
  + BMCA sections (sufficient for understanding what `ptp4l` does
  upstream; the library doesn't synthesise PTP messages itself).
- `~/docs/smpte/st2059-2-2021.pdf` — SMPTE PTP profile message rates,
  domain numbers, BMCA constraints, `defaultDS.slaveOnly` (the
  library inherits these via `ptp4l -f gPTP.cfg` or `2059.cfg`).
- `~/docs/smpte/st2059-1-2021.pdf` — SMPTE Epoch + signal-alignment
  math (consumed when the per-frame RTP-TS derivation lands in D3 —
  the SR anchor itself doesn't need the epoch math).
- `~/docs/ietf/rfc7273.txt` — `ts-refclk` / `mediaclk` grammar
  (already covered in Phase E10).
- `include/promeki/clockdomain.h` — `ClockEpoch::Absolute` was the
  right semantic for the new `ClockDomain::Ptp` ID.
- `man 2 clock_gettime` for the PHC clockid construction
  (`(~fd << 3) | 3`), `linux/ptp_clock.h` for `PTP_CLOCK_GETCAPS` /
  `PTP_SYS_OFFSET_PRECISE`, `man 2 adjtimex` for the kernel's
  TAI-UTC offset (`tx.tai`).

**What landed:**
- [x] **`PhcClock`** (new — `include/promeki/phcclock.{h,cpp}`).
      Userspace wrapper around `/dev/ptpN`.  Opens the device,
      snapshots capabilities via `PTP_CLOCK_GETCAPS`, reads the
      kernel's current TAI-UTC offset via `adjtimex`, exposes
      `now()` (TAI nanoseconds since Unix epoch as `TimeStamp`),
      `ntpNow()` (UTC NTP wallclock via TAI - leap-seconds shift),
      `sysOffsetPrecise()` (`PTP_SYS_OFFSET_PRECISE` cross-timestamp
      for sub-µs `(sys, phc)` mapping), and a heuristic `isLocked()`
      that flags sub-millisecond `|sysRealtime - phcUtc|` skew as
      "locked".  Move-only; the in-class `bindAsDomain` helper
      captures `(fd, taiUtcOffset)` by value rather than `this` by
      pointer so a subsequent move doesn't dangle the registered
      provider.  Linux-only — every reader returns
      `Error::NotSupported` on other platforms.
- [x] **`ClockDomain` extended.**  New well-known
      `ClockDomain::Ptp` ID (`ClockEpoch::Absolute`).  New
      `WallClockProvider` alias (`Function<int64_t()>` returning
      Unix-epoch UTC nanoseconds — universal wallclock representation
      so each domain's provider does its own TAI ↔ UTC conversion
      internally).  New static `setNowProvider` / `nowUtcNs` /
      `hasNowProvider` API plus instance-method convenience
      wrappers.  Registry mutations + reads serialise through a
      `Mutex` so a backend thread binding `Ptp` on open doesn't race
      a packetizer thread reading the provider.
- [x] **`RtpSession::setRtpAnchor(const ClockDomain &, uint32_t)`
      overload.**  Reads the domain's `nowUtcNs` and feeds it
      through `NtpTime::fromSystemClock`, producing an SR anchor
      that reflects the PTP timescale instead of `CLOCK_REALTIME`.
      Falls back to `NtpTime::now()` with a one-shot warn when the
      domain has no provider bound, so an application that opts in
      without actually wiring a clock degrades to today's behaviour
      rather than emitting a zeroed SR.
- [x] **`RtpMediaIO` PHC binding.**  New

**Composition with `RtpStreamClock` (unchanged):**
The SR-anchor → wallclock map (`RtpStreamClock`) is the receiver's
view; `RtpMediaClock` is the writer's view.  A PTP-aligned
deployment uses both: the writer computes RTP-TS via the media
clock and emits the SR's `(NTP, RTP-TS)` pair from the same
media-clock anchor; the receiver runs `RtpStreamClock` arithmetic
against the observed SR to derive wallclock for any subsequent
RTP-TS.  Receivers that bring their own `RtpMediaClock` (when the
PTP grandmaster is shared with the sender) can bypass the SR and
derive wallclock from any RTP-TS directly — that's the
`mediaclk:direct=0` interop case.

**Reading sweep complete (2026-05-21).** RFC 4175 read end-to-end;
ST 2110-20:2022 read cover-to-cover including Annex A; both code
paths (`RtpPayloadRawVideo::pack/unpack` and
`RtpMediaIO::configureVideoStream`) walked end-to-end.  Findings
that revise the checklist below:

- **All four code-side claims confirmed.** Single-SRD-per-packet (C
  always 0), F always 0, `extSeq` local to `pack()` (resets to 0 each
  call), no pixel-offset ascending check on receive.  `rtppayload.cpp`
  lines 162-264 (pack) and 267-298 (unpack) are the entirety of the
  RFC 4175 code today.
- **3-SRD hard limit (§6.2.1).** "RTP Packets shall not contain more
  than three Sample Row Data Headers."  Per-SRD overhead is 6 octets
  (Length 16 + F+Row 16 + C+Offset 16); ESN is 2 octets; full payload
  header for N SRDs is `2 + 6N`.  Cap N at 3.
- **SRD Length=0 keep-alive (§6.2.1).** When exactly one SRD Header
  is present, an SRD Length of 0 indicates no sample data follows.
  Lets a sender emit a frame-boundary marker packet without any
  payload — useful for low-rate streams that want to keep the
  receiver's frame-start detection synchronised.  Currently unused;
  E20a should accept it on read and expose a writer-side hook (no
  callers needed yet).
- **4:2:0 forbidden in PsF/interlaced (§6.2.5).** "The 4:2:0 sampling
  system shall only be applied to progressive scan images
  transmitted in a progressive manner.  This sampling system does
  not apply to PsF or interlaced video essence."  Hard reject at
  configure time (E20c / E20e).
- **4:2:0 progressive row pairing (§6.1.5 + §6.2.5).** SRD Row Number
  must be set to the **first luma row of the pair**; only every other
  Sample Row signaled.  The 4:2:0 progressive pgroup carries 8 luma
  + 2 chroma samples (Y'00, Y'01, Y'10, Y'11, C'B00, C'R00 for 8-bit
  / 4 pixels) and spans **two image rows** at once.  E20b's wire
  codec must understand this; E20a should treat 4:2:0 paired-row
  packetization as a separate code path under the same packetizer.
- **Marker bit semantics for interlaced (§6.1.2).** M=1 on the last
  packet of a **field**, not a frame.  Existing sender emits M on
  frame boundary only (rtppayload.cpp doesn't stamp marker — that's
  the session's job); needs an interlaced-aware override in E20e.
- **PsF F-bit semantics (§6.1.5).** For PsF data, the F bit indicates
  the **segment**, not the field — first segment F=0, second segment
  F=1.  SRD Row Number restarts at 0 at the top of each segment for
  PsF (same as interlaced fields).  Reception code must distinguish
  PsF from interlaced via the SDP `segmented` attribute.
- **PsF/interlaced odd-height split (§6.1.5).** If `height` is odd
  the temporally first field/segment carries one more line than the
  second.  E20e must split the source frame on this rule.
- **fmtp separator is `;` followed by whitespace (§7.1).** Existing
  `ImageDesc::toSdp` at `imagedesc.cpp:338-349` emits `;` with no
  whitespace.  E20c should switch the writer to `; ` (space-after-
  semicolon).  The parser already tolerates either form via
  `fmtpParameters()`.
- **`ImageDesc::toSdp` is the existing one-stop fmtp builder** for
  the raw branch (lines 320-349).  It already emits sampling, depth,
  width, height, colorimetry, RANGE — but is **not 2110-conformant**
  (emits `YCbCr-4:1:1` which is RFC 4175 only, not ST 2110-20; misses
  exactframerate / PM / SSN / TCS / MAXUDP / PAR / interlace /
  segmented).  E20c rewrites this path to dispatch ST 2110-20-eligible
  PixelFormats through a new `st2110video::fmtpToSdp` and leaves the
  legacy non-2110 RFC 4175 path (4:1:1) untouched.
- **Wire-format-conversion seam is in place.**
  `RtpMediaIO::_videoWirePixelFormat` + `UncompressedVideoPayload::
  convert()` is the existing seam where 8-bit YUYV → UYVY is swapped
  before packetization (`rtpmediaio.cpp:1585-1594`).  E20b extends
  this to the full ST 2110-20 wire-format conversion — the in-memory
  PixelFormat stays whatever the producer hands us; the wire format
  is the ST 2110-20 big-endian pgroup-packed octet stream.
- **PixelMemLayout coverage.** All 10/12/16-bit interleaved and
  planar layouts that ST 2110-20 maps to in-memory representations
  exist already (`pixelmemlayout.h:140-258`).  10-bit 4:2:2 Group A
  has `I_422_UYVY_3x10_BE`; 10-bit 4:4:4 has `I_3x10_DPX_B` (DPX
  Method B); 10-bit 4:2:0 has `SP_420_10_BE`.  E20b adds **wire-
  format** pgroup encoders, not new memory layouts.
- **The current `RtpMediaIO::configureVideoStream` raw branch rejects
  planar formats outright** (`rtpmediaio.cpp:1568-1572`: "planar pixel
  formats are not supported for RFC 4175 raw video (use an
  interleaved format)").  E20c must lift this for the 4:2:0 sampling
  + planar memory case — either by adding planar→wire packing in
  E20b or by converting through `UncompressedVideoPayload::convert()`
  to an interleaved memory format first.  Likely cleaner: have the
  wire packer accept a plane-major source.
- **The `bitsPerPixel = 8 * bytesPerBlock / pixelsPerBlock` formula**
  at `rtpmediaio.cpp:1573-1575` gives a whole-bits-per-pixel that
  matches the in-memory block size.  For ST 2110-20 the wire-format
  pgroup byte size differs from the in-memory block size for several
  rows (e.g. 10-bit 4:2:2 in-memory is `I_422_UYVY_3x10_BE` = 8 octets
  / 2 pixels for word-aligned LE / BE; wire is 5 octets / 2 pixels).
  The new packetizer must be parameterised by the **wire pgroup**,
  not the memory pgroup.

**E20c follow-on — CSC-driven wire-format conversion (E20c-2 / c-3 / c-4)**

The original E20c plan called for bit-level wire conversion *inside*
`RtpPayloadRawVideo`.  After E20c-1 landed, the architectural call was
to instead route the conversion through the existing CSC /
`PixelFormat` framework: every ST 2110-20 wire-format pgroup becomes
a first-class `PixelFormat` ID backed by a new `PixelMemLayout`, and
the existing `_videoWirePixelFormat` + `UncompressedVideoPayload::
convert()` seam at `rtpmediaio.cpp:1585-1594` drives the source→wire
conversion *before* packetization.  `RtpPayloadRawVideo` then becomes
a dumb byte-stride packetizer — no bit math at the packetizer layer,
no special-case 10/12-bit unpack inside the codec.  Benefits:

- One conversion path for every essence-format conversion, owned by
  CSC.  Same kernels are reusable for DPX / EXR / V210 file I/O,
  capture-card pixel formats, NDI / NTV2 / SDI write-back paths.
- Highway-accelerated fast paths fall out for free for the hot
  source→wire pairs (v210 → 2110/10, DPX-B → 2110/10, NV12 / P010 →
  2110/10).
- Source `PixelFormat` → wire `PixelFormat` is a CSC lookup; no
  parallel bit-pack table inside `st2110video`.
- The 4:2:0 paired-row SRD geometry stays a packetizer concern
  (which it always was per §6.2.5).  The wire format itself is
  conventional planar 4:2:0 with a specific bit packing — CSC can
  produce that without knowing anything about RTP.

**E20c-3 part C — 4:2:0 architecture landed 2026-05-21.**

The 4:2:0 wire format does **not** fit the existing CSC per-row
iterator model.  ST 2110-20 §6.2.5 requires the SRD payload to be
a single packed pgroup stream where each pgroup carries 4 Y samples
(2 from row N + 2 from row N+1) + 1 Cb + 1 Cr, spanning **two
image rows per pgroup**.  The `LineFuncPtr` signature (kernel called
per source image row) cannot assemble a pgroup that needs both
rows of a pair at once — the kernel for y=0 writes Y from row 0
into the pgroup's Y slot and then the kernel for y=1 would
overwrite the wire bytes it shares with the first row's pgroup.

Two viable architectures:

1. **Per-row-pair iterator** — extend `CSCPipeline::execute` to
   recognise destination layouts with `Sampling420` and iterate
   one pair (2 source rows → 1 wire row) at a time.  Cleanest at
   the kernel level: each fast path call receives 2 source row
   pointers and produces a complete row of 4:2:0 wire pgroups.
   Requires a new `LineFuncPtr2` signature or a side-channel that
   passes "rowsPerCall" to the fast path.
2. **Semi-planar wire layout + packetizer interleave** — keep the
   wire layout as the current `SP_420_*_BE_2110` (two-plane Y +
   interleaved CbCr, both bit-packed).  The CSC kernel converts
   source planes to wire planes per row (Y per source row, CbCr
   per pair).  `RtpPayloadRawVideo` learns to read from two wire
   planes and interleave Y / CbCr into pgroups when emitting SRDs.
   The wire byte stream that flows through the packetizer is no
   longer a flat single-plane buffer.

Option 1 keeps `RtpPayloadRawVideo` simple but requires a CSC
framework change.  Option 2 keeps CSC simple but requires the
packetizer to gain multi-plane awareness.

**Architecture (landed):** Option 1.  The 4:2:0 wire layouts
(`I_420_BE_2110_8/10/12`) are single-plane with `vSubsampling=2` on
plane 0 — one wire byte row carries pgroups spanning 2 image rows.
`CSCPipeline::execute` detects `dstDesc.memLayout().planeDesc(0).vSubsampling > 1`
and steps the outer loop by that amount; the kernel reads the
second source luma row internally via `srcStrides[0]`.  The
`SP_420_*_BE_2110` IDs (added in c-2 with the wrong semi-planar
model) are replaced by the single-plane variants.  Also added
`I_420_BE_2110_8` (= 93) and `YUV8_420_2110_Rec709` (= 231) since
8-bit 4:2:0 wire is NOT byte-identical to NV12 (NV12 is row-major
planar; ST 2110 wire is pgroup-interleaved per §6.2.5).

  `ceil(height/vSub)` wire rows).

**Packetization (§6.1.4 - §6.3):**
- [x] RTP Payload Header = Extended Sequence Number (16 bits, high 16
      of a per-stream 32-bit extended seq) + 1-3 SRD headers (hard
      cap at 3 per §6.2.1).  **Landed 2026-05-21 in E20a.**

      assertion in `pack()`.

      the same packet. Required to avoid sub-1000-octet packets in GPM

      asserts the rule end-to-end.
- [x] **F (Field) bit** — interlaced: 0 = first field, 1 = second

```
N_PACKETS = packets per frame
T_FRAME   = 1 / frame_rate
R_ACTIVE  = active-time ratio (1080/1125 progressive, etc.)
T_RS_g    = T_FRAME * R_ACTIVE / N_PACKETS   (gapped)
T_RS_l    = T_FRAME / N_PACKETS              (linear)
TPR_j     = j * T_RS + T_VD
T_VD      = N * T_FRAME + TR_OFFSET          (N integer, T_VD on
                                              SMPTE Epoch)
```

- [x] **Type N (gapped, narrow)** —
      `VRX_FULL = max(1500*8 / MAXUDP, N_PACKETS / (27000 * T_FRAME))`,
      `CMAX = max(4, N_PACKETS / (43200 * R_ACTIVE * T_FRAME))`,
      `β = 1.10`. Signal `TP=2110TPN`.  Landed 2026-05-22.
      `St2110Tx::vrxFullNarrowBytes` / `cmaxNarrowPackets` /
      `tpFmtpValue(TypeN)`.  The TX-side VideoTxThread stamps
      per-packet TPR_j via the gapped PRS when the operator
      explicitly pins TypeN (Auto only resolves to TypeNL because
      true Type N requires DPDK).
- [x] **Type NL (linear, narrow)** — same VRX_FULL / CMAX formulas as
      Type N but uses the linear PRS. Signal `TP=2110TPNL`.  Landed
      2026-05-22.  `St2110Tx::trsLinear` drives the per-packet TPR_j
      stride; `St2110Tx::resolveSenderType(TxTime)` returns TypeNL
      so a host with PHC-bound `RtpPacingMode::TxTime` self-labels
      as NL automatically.
- [x] **Type W (linear, wide)** —
      `VRX_FULL = max(1500*720 / MAXUDP, N_PACKETS / (300 * T_FRAME))`,

      Signal `TP=2110TPW`.  Landed 2026-05-22.
      `St2110Tx::vrxFullWideBytes` / `cmaxWidePackets`.
      `resolveSenderType(KernelFq)` and `resolveSenderType(Userspace)`

- [x] `TROFF` Format Specific Parameter when TR_OFFSET ≠

      `RtpVideoTrOffsetUs` / `RtpAudioTrOffsetUs` / `RtpDataTrOffsetUs`

**Receivers (§7.2):**

ST 2110-21 receivers are classified Type N / W / A (asynchronous).
For us the receiver is the depacketizer; we accept all three by
default (Type A) because we don't enforce ts-refclk identity at the
receiver — that's a deployment policy, not a wire-level constraint.
- [ ] Surface receiver type via `executeCmd(Params)` so test fixtures

### Phase E22 — ST 2110-22 Compressed Video (groundwork)

**Status:** landed 2026-05-22.  Full SDP wrapper + K=1 slice-mode

**Encoder PD-selection follow-ups landed 2026-05-22:**

Developing a `rtp.jxs.*` end-to-end loopback case in `promeki-test`
surfaced four interrelated bugs in the encoder-side plumbing that
prevented the planner from honouring a user-configured compressed
output `PixelFormat`:

- `JpegXsVideoEncoder::supportedInputList` listed non-planar IDs
  (`YUV10_422_Rec709` etc.) that the encoder's internal
  `classifyInput` rejected, causing the planner to pick a TPG output
  format the encoder then refused at runtime.  Now mirrors
  `classifyInput`'s planar set exactly.
- `VideoEncoderMediaIO::describe`, `proposeOutput`, and
  `executeCmd(Open)` all picked `_codec.compressedPixelFormats()[0]`
  as the producible / preferred / runtime output PD, ignoring a
  user-configured `VideoPixelFormat`.  For chroma-/depth-agnostic
  codecs (`H264`, `HEVC`) the registry's first PD happened to match
  the natural default, but for chroma-/depth-specific families
  (`JPEG_XS_*`) the first-in-registry pick silently disagreed with
  the configured output and the SDP emitted the wrong depth.  All
  three sites now honour a configured `VideoPixelFormat` when it's a
  compressed PD for the active codec.
- `VideoEncoderMediaIO::proposeInput` defaulted `wantChroma` to 4:2:0
  via the `VideoChromaSubsampling` spec default, which then forced
  the planner to swap a directly-supported upstream PD onto a
  bogus 4:2:0 variant.  Now short-circuits: if the upstream PD is
  already in the encoder's supportedInputs list, accept it as-is
  without re-querying the chroma heuristic.
- `JpegXsVideoDecoder::submitFrame` left its persistent SVT decoder
  context in place after a `decoder_get_frame` failure, so a single
  malformed bitstream wedged every follow-on frame.  Now calls
  `_impl->closeIfOpen()` on failure so the next bitstream re-parses
  its picture header from scratch.

These all land in this changeset.  An e2e loopback case driving the
full TPG → encoder → RTP → depacketizer → decoder → Inspector chain
is **deferred** — promeki-test exposes a separate RX-side issue
where a startup-window UDP loopback packet loss corrupts the first
frame's codestream header, which SVT-JPEG-XS treats as a fatal
condition.  Recovery on the decoder side is in place; the upstream
trigger (startup packet-loss at high packet rates) is the remaining
gap.

**Wire format (§5.4):**
```
+---+---+---+---+--------+-------------------------+
| 0 | 0 | B | F | PCUV   | DATA24                  |
+---+---+---+---+--------+-------------------------+
   1   1   1   1    4              24
B = 1 on first subframe of AES3 Block (Z preamble).
F = 1 on first subframe of AES3 Frame (X preamble) or first subframe
    after Z; 0 = second subframe (Y preamble).
PCUV = AES3 P/C/U/V bits (timeslot 31 / 30 / 29 / 28).
DATA24 = 24 audio bits from AES3 timeslots 4..27, big-endian.
```

**Checklist:**
- [ ] `RtpPayloadAm824` — pack N AM824 subframes per packet at ptime ∈
      {0.08, 0.12, 0.14, 0.09, 1, 1.09} ms depending on sample rate.
- [ ] Preserve B (block start), F (frame start), P/C/U/V on round-trip.
- [ ] RTP-TS = sample at the X or Z preamble of the first AES3 Subframe
      in the packet (§5.3).
- [ ] SDP: `m=audio <port> RTP/AVP <pt>` + `a=rtpmap:<pt>

- [ ] Permitted (rate, ptime) pairs per Table 1 (§6.1) — enforce on
      configure.
- [ ] Channel-order extension (§6.2): `AES3` grouping symbol added on

- [ ] Hand off the SMPTE 337-decoded `Dolby E` / `AC-3` payloads to

### Phase E2022 — ST 2022-7 Seamless Protection Switching

**Status:** landed 2026-05-22.  TX fan-out, RX dual-leg listener with
seq-aligned dedup, full SDP `a=group:DUP` / `a=mid` round-trip, and
the new per-leg `MediaConfig` bind-address + interface keys all in.
End-to-end functional loopback test is deferred to a follow-up; the
unit tests cover the core mechanics (same seq/SSRC across legs,
secondary failure does not break primary, SDP round-trip).

**Required reading (done as part of the landing):**
- `~/docs/smpte/st2022-7-2019.pdf` — receiver "shall accept" wording;
  timing-window tolerance (50 ms recommended); duplicate detection
  via RTP sequence number.  The library leverages the existing
  `RtpSeqReorderBuffer`'s silent duplicate-discard for dedup — no
  new policy needed.
- `~/docs/ietf/rfc7104.txt` — SDP DUP semantics; primary/secondary
  mids are paired in the session-level `a=group:DUP` value.
- `~/docs/ietf/rfc5888.txt` — SDP group/mid grammar.
- `src/network/rtpseqtracker.cpp` and `rtpseqreorderbuffer.cpp` —
  the reorder buffer already does dedup-by-extended-seq via
  `droppedAsDuplicate`; no changes needed.

**Architecture (landed):**
One packetizer, two transports (refcounted RTP packet buffers shared
across both legs — zero copy on fan-out).  Each transport runs its
own `PacketScheduler` thread so back-pressure on one leg cannot
stall the other; both schedulers compute TPR_j from the same media
clock so they agree on send times without cross-leg locking.  The
receive side runs one `ReceiveThread` per leg; both feed the same
`StreamReceiver` list and the per-leg dispatch is serialised by a
single `_dispatchMutex` so the seq tracker / SSRC pin / reorder
buffer remain coherent under dual-leg recv.  RFC 7104 dedup
mechanics ride on the existing reorder buffer's silent discard of
already-buffered / already-emitted extended seqs.

## Appendix C — Stream health surface

A single composite "is this stream healthy" boolean for monitoring /
ops, derived from the per-stream stats already collected.

**A writer-mode stream is healthy when all of:**
- The transport is open and the per-stream TX thread is running.
- The most-recent emission is recent (< 2 × frame interval for video,
  < 5 × packet time for audio).
- The pacing scheduler has no overrun queue depth > VRX_FULL.
- The bound `ClockDomain` reports `Locked` (PTP slave is in sync, or
  the local-MAC fallback is configured deliberately).
- The companion RTCP scheduler is emitting SRs.
- For 2022-7 senders: both legs report wire activity within the same
  bound.

**A reader-mode stream is healthy when all of:**
- At least one SR has been observed within `10 × RtcpInterval`
  (typically 50s).
- No SSRC change observed in the last 5s.
- `RxInterarrivalJitter` (RFC 3550 §A.8) is below a per-stream
  threshold (default: video = 3 × T_RS, audio = 100 µs, ANC = 1 ms).
- `RxReorderDroppedOverflow` and `RxReorderDroppedDuplicate` did not
  advance in the last second.
- `RxFramesDroppedSsrcReset` did not advance in the last second.
- `lastPacketArrivalNs` is within 5 × frame interval (or audio
  cadence) of now.
- For 2022-7 receivers: both legs are receiving packets within 100 ms
  of each other; either leg satisfying the above keeps the composite
  healthy.

**Exposure:**
- `sendParams(MediaIOParams().get(ParamHealth))` fills the action's
  value with a `bool` health flag; `sendParams(MediaIOParams().get(ParamHealthReason))`
  fills with a `String` reason tag (empty when healthy, else e.g.
  `"NoSrObserved"`, `"SsrcReset"`, `"PtpUnlocked"`).
- `MediaIOStats::ID StreamHealth` (bool) for trend logging.
- `MediaIOStats::ID StreamHealthReason` (String) — the latest reason
  string (empty when healthy).

This is pure composition over existing counters. The thresholds are
configurable via new keys: `RtpHealthMaxJitterUs`,
`RtpHealthSrTimeoutMs`, `RtpHealthSilenceTimeoutMs`. Defaults match
the per-stream specifics above.

## Appendix D — Glossary

The ST 2110 family is acronym-dense, especially -21 and -40. Cross-
referenced for quick lookup while reading the plan.

**Timing (mostly ST 2110-21 + -40):**
- **T_FRAME** — frame period (1 / frame rate).
- **T_LINE** — line period (T_FRAME / lines per frame) for
  interlaced / PsF systems.
- **T_VD** — Video Transmission Datum; integer multiples of T_FRAME
  shifted by TR_OFFSET, anchored to the SMPTE Epoch (ST 2059-1).
- **TR_OFFSET** — sender's shift of T_VD from N × T_FRAME.
- **TRO_DEFAULT** — sender-type's default TR_OFFSET; values per
  -21 §6.3 / §6.4.
- **T_RS** — Time Read Spacing; spacing between adjacent VRX packet
  reads.
- **TPR_j** — Packet Read Time j; `T_VD + j × T_RS`.
- **TPR_0** — first packet's read time (= T_VD).
- **R_ACTIVE** — ratio of active scan time to total frame time
  (1080/1125 for 1080p, etc.).
- **R_NOMINAL** — long-term average packet rate
  (N_PACKETS / T_FRAME).
- **N_PACKETS** — packets per frame for -20 / -22 streams.
- **T_DRAIN** — VRX bucket drain interval (T_FRAME / N_PACKETS) / β.
- **β (beta)** — VRX rate scaling factor; 1.10 for all -21 types.

**Buffer / shaping model (ST 2110-21):**
- **VRX** — Virtual Receiver Buffer (leaky-bucket model on the
  sender).
- **VRX_FULL** — VRX bucket capacity, in packets, per sender type.
- **C_INST** — instantaneous VRX occupancy.
- **C_MAX** — per-type ceiling for C_INST.
- **PRS** — Packet Read Schedule; gapped (Type N) or linear
  (Types NL / W).

**Sender / receiver types:**
- **Type N** — narrow, gapped — high-end ST 2110 sender.
- **Type NL** — narrow linear — distributes the gap.
- **Type W** — wide — software-friendly, larger CMAX.
- **Type A (rx)** — asynchronous receiver — tolerates everything.

**Delay (ST 2110-10 §7.8 - §7.9):**
- **D_LO** — Link Offset Delay (T_REC − T_RTP); receiver's playout
  budget.
- **D_TX** — Transmission Delay (T_TX − T_RTP); sender's egress
  delay.
- **D_NET** — Network Delay.
- **D_PROC** — internal processing delay through a
  receive-process-send device.

**SDP fmtp values:**
- **MAXUDP** — Standard (1460) or Extended (≤8960) UDP datagram
  size.
- **TSMODE** — RTP timestamp mode:
  - **SAMP** — RTP-TS reflects original sample instant.
  - **NEW** — RTP-TS created anew at egress.
  - **PRES** — RTP-TS preserved from input that did not signal SAMP.
- **TSDELAY** — sender's D_TX in microseconds.
- **TP** — sender Type fmtp (2110TPN / 2110TPNL / 2110TPW).
- **TROFF** — TR_OFFSET fmtp (microseconds, video) or 90 kHz ticks
  (ANC).
- **TM** — ANC Transmission Model (LLTM / CTM) — ST 2110-40 §6.4.
- **SSN** — SMPTE Standard Number fmtp (e.g. `ST2110-20:2022`).
- **PM** — Packing Mode (`2110GPM` / `2110BPM`).
- **GPM** — General Packing Mode (any pgroup-aligned).
- **BPM** — Block Packing Mode (180-octet block multiples).

**Wire format (ST 2110-20):**
- **SRD** — Sample Row Data; per-line segment in the RFC 4175 payload
  header.
- **ESN** — Extended Sequence Number; high 16 bits of the 32-bit
  packet sequence (low 16 in the RTP header).
- **pgroup** — minimum byte-aligned sample group.
- **F bit** — Field Indication (0 = first field / progressive,
  1 = second field).
- **C bit** — Continuation (another SRD header follows in this
  packet).

**Ancillary (ST 2110-40 §6):**
- **T_FST** — Transmitted Frame Start Time.
- **T_SST** — Transmitted Second-Field Start Time.
- **T_EPO** — Earliest Packet Time Offset.
- **T_LBO** — Location-Based Time Offset.
- **T_SFO** — Second Field Offset.
- **T_D** — sender's per-packet delay budget (LLTM = 8 /
  (rate × totalLines); CTM = 1 ms).
- **LLTM** — Low-Latency Transmission Model.
- **CTM** — Compatible Transmission Model.
- **VPID_Code** — Video Payload ID (RFC 8331 / ST 352).

**Clock (RFC 7273 + ST 2110-10):**
- **Common Reference Clock** — facility-wide PTP-derived clock.
- **Timestamp Reference Clock** — what `ts-refclk` names.
- **Media Clock** — what `mediaclk` names; matches the essence
  sample rate.
- **RTP Clock** — the counter actually placed in the RTP header.
- **ts-refclk** — SDP attribute for the reference clock
  (`ptp=...` / `localmac=...` / `ptp=...:traceable`).
- **mediaclk** — SDP attribute for the media clock
  (`direct=<offset>` / `sender`).
- **CNAME** — RTCP SDES item identifying the endpoint.

**Redundancy (ST 2022-7 + RFC 7104 + RFC 5888):**
- **DUP** — Duplication grouping semantics.
- **MID** — Media Identification (`a=mid:...`).

**AES3 / -31 (deferred):**
- **AM824** — AES3 subframe envelope used by -31.
- **PCUV** — AES3 P/C/U/V status bits.
- **DATA24** — AES3 24-bit audio payload.
- **AES3 Block** — 192 frames.
- **AES3 Frame** — 2 subframes.
- **AES3 Subframe** — one channel sample plus preamble + status.

## Open questions / decisions

- **Type self-labelling on Linux**: do we ever self-claim Type N
  without DPDK or NIC TXTIME? I lean **no** — claim Type W when
  paced via kernel fq, Type NL only when the bench measurement
  proves it (and only with SCM_TXTIME on supported NICs).
- **Extended UDP Size Limit (8960 octets)**: useful for high-bitrate
  4K/8K -22 over jumbo Ethernet, breaks anything that crosses a
  router. Hide behind an explicit opt-in
  (`MediaConfig::RtpExtendedUdpEnabled`).
- **One stream per session vs. session multiplexing**: ST 2110-10
  §6.2 **forbids** RTP session multiplexing on the same multicast
  group/port. We already enforce this implicitly (one
  `UdpSocketTransport` per stream). Add an explicit guard at config
  validation.
- **PTP slave dependencies**: linuxptp's `ptp4l` is the obvious
  reference; we can either embed a slave (significant work) or rely
  on a host-running `ptp4l` and read `/dev/ptp0` via `PHC` ioctls.
  The latter is the pragmatic Phase D2 default; embedded PTP is a
  far-future option.
