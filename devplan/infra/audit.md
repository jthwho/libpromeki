## Categories

- `std` â€” coding-standard compliance (header layout, naming, indentation,
  type-alias conventions, no-`bool`-error-returns, etc.).
- `doc` â€” Doxygen completeness and correctness; thread-safety statements.
- `api` â€” ergonomics, extensibility, consistency.
- `bug` â€” correctness, lifetime, threading, missing error paths.
- `log` â€” `PROMEKI_DEBUG` registration, log-message context, severity.
- `test` â€” public API not covered by unit tests (audit notes only;
  test-coverage-enforcer addresses these).

## Severity scale

- `crit` â€” known or near-certain bug, security issue, undefined behavior.
- `high` â€” likely bug, broken contract, missing critical error path,
  thread-safety hazard.
- `med` â€” coding-standard or doc problem on a non-trivial API.
- `low` â€” minor style, doc nit, cleanup opportunity.
- `idea` â€” feature or ergonomics suggestion (no defect implied).

## Remediation phases

Findings are scheduled into prioritized chunks:

- **R1** â€” `crit` and `high` bug fixes. **COMPLETE** (R1aâ€“R1l, 2026-04-26).
- **R2** â€” `high` API contract fixes (missing error returns, broken
  thread-safety, raw `new`/`delete` left in code).
- **R3** â€” Standards-compliance sweep: rename, reorder, doc, header
  layout. Mostly mechanical; can be batched per module.
- **R4** â€” Ergonomics/extensibility: missing operators, missing aliases,
  missing wrappers.
- **R5** â€” Documentation pass: any remaining `doc` findings, plus a
  thread-safety statement on every class that lacks one.
- **R6** â€” Test-coverage backlog (`test`-tagged findings) handed to
  test-coverage-enforcer.
