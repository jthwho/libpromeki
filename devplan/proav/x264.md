# x264 H.264 Software Encoder Backend

**Library:** `promeki` (gated by a single `PROMEKI_ENABLE_X264` flag)
**Backend name:** `x264` (registered against `VideoCodec::H264`)
**Scope:** **Encode only.** libx264 is an encoder; it has no decoder.
**Status:** Planning.
**Standards:** All work follows `CODING_STANDARDS.md`; every host-testable
helper requires unit tests (doctest). Build must be warning-free.

> ⚠️ **Licensing.** libx264 is **GPL-2.0-or-later** (or a paid commercial
> license from x264 LLC). Linking it into `libpromeki.so` makes that build
> of the shared object subject to the GPL. It is gated behind
> `PROMEKI_ENABLE_X264` (**default ON** per owner direction) and vendored as
> a separate shared object shipped alongside `libpromeki.so` (the same model
> used for the LGPL `libmp3lame` / `libmpg123`), which keeps the dynamic
> boundary clean and lets an end user drop in their own libx264 build.
> Because it is on by default, the build emits a **GPL-compliance notice** at
> configure time via a reusable mechanism (§4.5) that libx265 and any future
> GPL dependency will share. `THIRD-PARTY-LICENSES` carries the x264 entry
> and a dedicated GPL-components section.

---

## 1. Goal

Add a vendored, cross-platform (x86-64 **and** ARM/aarch64) software H.264
encoder that plugs into the existing `VideoEncoder` / `VideoCodec` registry
as the backend named `x264`, so that:

```cpp
MediaIO::Config cfg;
cfg.set(MediaConfig::Type,       "VideoEncoder");
cfg.set(MediaConfig::VideoCodec, "H264");       // unpinned, or…
cfg.set(MediaConfig::CodecBackend, "x264");     // …pin the software backend
// or simply VideoCodec("H264:x264")
```

works end-to-end through `VideoEncoderMediaIO` with **no MediaIO-layer
changes** — exactly as `JpegXsVideoEncoder` ("Svt") and `NvencVideoEncoder`
("Nvidia") already do.

The encoder must expose the pro-video surface: **8/10-bit**, **4:2:0 / 4:2:2
/ 4:4:4**, **lossless**, **HDR static metadata** (mastering-display + CLL
SEI), **CEA-708 caption SEI** (ANC → `user_data_registered_itu_t_t35`),
**SMPTE timecode SEI**, and **VUI colour signalling** — all driven from the
generic `MediaConfig` keys that already exist.

### Why encode-only

`libx264` does not decode. The codebase already has hardware H.264 decode
(`NvdecVideoDecoder`). A full-featured *software* H.264 decoder (covering
Hi10 / Hi422 / Hi444) would require a different library (ffmpeg's
`libavcodec`) and is explicitly **out of scope** for this work. The `x264`
backend registers an encoder factory only; `VideoCodec::H264.canDecode()`
stays driven by whatever decoder backends are present.

## 2. Architecture fit (what already exists)

Everything below is already in place and is reused unchanged:

| Concern | Existing facility |
|---|---|
| Codec family + backend handles | `VideoCodec` / `VideoCodec::Backend`, `VideoCodec::H264` (`videocodec.h`) |
| Encoder session base + registry | `VideoEncoder` (`videoencoder.h`): `submitFrame`/`receiveFrame`/`flush`/`reset`, `BackendRecord`, `registerBackend`, `selectInputPayload`, `selectAncForSei`, `buildOutputFrame` |
| MediaIO wrapper | `VideoEncoderMediaIO` ("VideoEncoder") — picks codec via `MediaConfig::VideoCodec`, backend via `MediaConfig::CodecBackend` |
| Compressed output | `CompressedVideoPayload` (`PixelFormat::H264` = 135), `FrameType` (`enums_mediaio.h`) |
| Generic encoder config keys | `MediaConfig`: `BitrateKbps`, `MaxBitrateKbps`, `VideoRcMode`, `GopLength`, `IdrInterval`, `BFrames`, `LookaheadFrames`, `VideoPreset`, `VideoProfile`, `VideoLevel`, `VideoQp`, `VideoRepeatHeaders`, `VideoTimecodeSEI`, `VideoSeiCaptionsEnabled`, `VideoColorPrimaries/Transfer/Matrix/Range`, `VideoScanMode`, `HdrMasteringDisplay`, `HdrContentLightLevel`, `FrameRate`, `Capacity` |
| Output metadata keys | `Metadata::CodecParameterSets`, `CodecFrameAvgQP`, `CodecEncodeOrderIdx`, `CodecDisplayOrderIdx`, `CodecGopPosition`, `VideoScanMode`, … |
| H.264 bitstream framing | `H264Bitstream` (`h264bitstream.h`): Annex-B/AVCC walk + convert, `parseSpsResolution`; `AvcDecoderConfig` SPS/PPS extraction |
| ANC → SEI translation | `AncTranslator::translate(pkt, AncTransport::HlsSei)` for CEA-708 |
| Input pixel formats | `PixelFormat::YUV{8,10}_{420,422,444}_Planar[_LE]_Rec709` (and Rec2020 PQ/HLG planar HDR variants) |

The `x264` backend is therefore a **single new translation unit** plus a
**foundation/refactor phase** that lifts reusable logic out of NVENC.

## 3. Foundation phase — generalize shared H.264 logic first

Per the library-first / foundation-first preference, before writing the
backend we extract the codec-agnostic pieces currently trapped inside
`NvencVideoEncoder`'s anonymous namespace / `Impl` into reusable library
objects. NVENC is refactored to consume them (behaviour-preserving), and
x264 consumes them from day one. Each item ships with its own unit tests.

### 3.1 H.273 resolution — folded onto `PixelFormat` ✅ landed
`NvencVideoEncoder::Impl::resolveColorDescription()` +
`populateVuiColorDescription()` were lifted onto `PixelFormat` itself
(`pixelformat.h` / `pixelformat.cpp`):

```cpp
static constexpr uint32_t PixelFormat::ColorAuto = 255;
struct PixelFormat::H273ColorDescription { uint32_t primaries, transfer, matrix, range; };
struct PixelFormat::VuiColorSignal { bool signalTypePresent, fullRange, colorDescriptionPresent; uint32_t primaries, transfer, matrix; };

// Resolves ColorAuto(255)/unset(0)/explicit overrides against this format's
// ColorModel::toH273() + videoRange().
H273ColorDescription PixelFormat::resolveH273(uint32_t cfgPrim = ColorAuto, ...) const;
// Pure VUI present/normalise decision from a resolved tuple.
static VuiColorSignal PixelFormat::vuiColorSignal(const H273ColorDescription &);
```

NVENC keeps its struct→`NV_ENC_*` copy but sources the numbers here. x264
writes the same numbers into `x264_param_t.vui.{i_colorprim, i_transfer,
i_colmatrix}` and `vui.b_fullrange`.

### 3.2 `VideoEncoderSei` helper — SEI payload assembly ✅ landed (captions + HDR)
**`include/promeki/videoencodersei.h` + `.cpp`** (kept standalone — SEI
assembly is genuinely encoder-specific, not a `PixelFormat`/`ColorModel`
concern). Landed: `captions()` (type 4), `masteringDisplay()` (137),
`contentLightLevel()` (144), each returning a `SeiPayload {int type; Buffer
bytes;}`; NVENC now consumes `captions()`. The pic_timing/timecode builder
(§6.7) remains deferred to the x264 phase.

Both NVENC and x264 attach SEI as a list of `(payloadType, bytes)` — NVENC
via `NV_ENC_SEI_PAYLOAD`, x264 via `x264_sei_payload_t`. The
backend-neutral collector:

```cpp
struct SeiPayload { int type; Buffer bytes; };   // type = H.264/HEVC SEI payloadType

// Caption SEI (payloadType 4): walks the source Frame's ANC via
// VideoEncoder::selectAncForSei(frame, streamIdx, {AncFormat::Cea708}),
// runs each packet through AncTranslator → HlsSei, returns the
// user_data_registered_itu_t_t35 payload bytes. Lifted verbatim from
// the NVENC caption block (nvencvideoencoder.cpp:650-704).
List<SeiPayload> buildCaptionSei(const Frame &source, int videoStreamIndex);

// NEW raw-byte RBSP builders (NVENC gets these from hardware; x264 must
// supply the bytes itself):
Buffer buildMasteringDisplaySei(const MasteringDisplay &);   // payloadType 137
Buffer buildContentLightLevelSei(const ContentLightLevel &); // payloadType 144
Buffer buildPicTimingTimecodeSei(const Timecode &, VideoScanMode); // payloadType 1 (see §6.7 caveat)
```

NVENC is refactored to call `buildCaptionSei()` (replacing its inline
loop). The HDR/timecode byte-builders are new and used by x264 (NVENC
continues to signal those through the driver).

### 3.3 `H264Profile` TypedEnum + `H264ProfileLevel` wire layer ✅ landed
- **`H264Profile`** is now a first-class `TypedEnum` in `enums_codec.h`
  (`Auto`/`Baseline`/`Main`/`High`/`High10`/`High422`/`High444`/
  `ProgressiveHigh`, default `Auto`) — validation / display / `toString`
  for free.
- **`H264ProfileLevel`** (`h264profilelevel.h/.cpp`) is the
  serialization layer between the lowercase x264/ffmpeg config wire tokens
  and the enum: `profileFromWire(String)→H264Profile`,
  `profileToWire(H264Profile)→String` (for `x264_param_apply_profile`),
  `autoProfile(chromaIDC, bitDepth)→H264Profile`, and `levelIdc(String)→int`
  (`"4.1"`→41, `"1b"`→9, ""→0 auto).

`MediaConfig::VideoProfile` stays a codec-agnostic **String** (HEVC/AV1
have their own vocabularies). NVENC maps `profileFromWire` → GUID; x264
uses `profileToWire` for `apply_profile` and `levelIdc` for
`i_level_idc`. (An unrecognised token falls back to `Auto` — geometry-
derived — since `Enum::isValid()` only checks the type, not the value.)

### 3.4 Geometry accessors — folded onto `PixelFormat` ✅ landed
Rather than a standalone struct, the promeki-side geometry is now
first-class on `PixelFormat`:

```cpp
int PixelFormat::chromaFormatIDC() const;   // 0=mono,1=4:2:0,2=4:2:2,3=4:4:4
int PixelFormat::bitsPerComponent() const;  // 8/10/12 (luma component)
int PixelFormat::bytesPerComponent() const; // ceil(bits/8)
// + existing planeCount(); uvHeightDivisor is (chromaFormatIDC==1 ? 2 : 1)
```

x264's `X264_CSP_*` table reads these directly; the "is this 4:4:4 /
10-bit?" lossless/profile-auto checks use `chromaFormatIDC()==3` /
`bitsPerComponent()>8`. NVENC keeps its `NV_ENC_BUFFER_FORMAT`-coupled
`FormatEntry` table (hardware-enum-specific).

### 3.5 `stampEncodedPacketMetadata()` — output metadata
**Optional, lower priority.** The `Metadata`-stamping tail of
`lockAndBuildPacket()` (`nvencvideoencoder.cpp:1707-1772`) — keyframe/
intra-refresh/discardable flags, `FrameType`, GOP position, scan mode,
duration-from-framerate, `CodecParameterSets` — is largely generic. Factor
a helper taking a small POD of per-frame values so both backends stamp
identically. Can land with x264 or as a follow-up.

> Refactor discipline: each extraction is behaviour-preserving for NVENC,
> lands with NVENC still green (its functional test must stay 92/0/2), and
> carries new unit tests for the lifted helper.

## 4. Vendoring libx264 (cross-platform)

### 4.1 Submodule
Add `thirdparty/x264` as a git submodule:

```
[submodule "thirdparty/x264"]
    path = thirdparty/x264
    url  = https://code.videolan.org/videolan/x264.git
```

(Pin to a stable release tag, e.g. a known-good `stable` commit.)

### 4.2 Build wiring — autotools-style `./configure`, like lame/mpg123
x264 uses its own shell `./configure` + `make` (not CMake), producing
`libx264.so`. Model the `ExternalProject_Add` on the **liblame** block
(`CMakeLists.txt:2074`), building **in-source into a copy** and installing
into `${PROMEKI_THIRDPARTY_PREFIX}`:

```cmake
# --- x264 (GPL H.264 encoder) ---
if(PROMEKI_ENABLE_X264)
    if(PROMEKI_USE_SYSTEM_X264)
        find_package(PkgConfig REQUIRED)
        pkg_check_modules(X264 REQUIRED IMPORTED_TARGET x264)
        set(PROMEKI_X264_TARGET PkgConfig::X264)
    else()
        set(PROMEKI_X264_SRC_COPY ${CMAKE_CURRENT_BINARY_DIR}/thirdparty-src/x264)
        ExternalProject_Add(
            libx264-ep
            DOWNLOAD_COMMAND ${CMAKE_COMMAND} -E copy_directory
                ${CMAKE_CURRENT_SOURCE_DIR}/thirdparty/x264 ${PROMEKI_X264_SRC_COPY}
            SOURCE_DIR ${PROMEKI_X264_SRC_COPY}
            BUILD_IN_SOURCE 1
            CONFIGURE_COMMAND
                ${CMAKE_COMMAND} -E env "CFLAGS=-fPIC -O2"
                ${PROMEKI_X264_SRC_COPY}/configure
                    --prefix=${PROMEKI_THIRDPARTY_PREFIX}
                    --libdir=${PROMEKI_THIRDPARTY_PREFIX}/lib
                    --enable-shared --disable-static --enable-pic
                    --disable-cli --disable-opencl --disable-swscale --disable-lavf
                    --bit-depth=all          # single lib supports 8- AND 10-bit at runtime
                    --chroma-format=all      # 4:2:0 + 4:2:2 + 4:4:4
                    ${_PROMEKI_X264_CROSS_ARGS}   # --host=/--cross-prefix= when cross-compiling
            BUILD_COMMAND make
            INSTALL_COMMAND make install
            BUILD_BYPRODUCTS ${PROMEKI_THIRDPARTY_PREFIX}/lib/libx264.so
        )
        add_library(promeki_x264_vendored SHARED IMPORTED)
        set_target_properties(promeki_x264_vendored PROPERTIES
            IMPORTED_LOCATION ${PROMEKI_THIRDPARTY_PREFIX}/lib/libx264.so)
        add_dependencies(promeki_x264_vendored libx264-ep)
        set(PROMEKI_X264_TARGET promeki_x264_vendored)
    endif()
endif()
```

Then, alongside the other codecs near `CMakeLists.txt:3391`/`3798`:
gate the source file, add the include dir, link `${PROMEKI_X264_TARGET}`,
and `target_compile_definitions(promeki PRIVATE PROMEKI_ENABLE_X264)`. Add
`libx264.so` to the staged-shared-deps + `$ORIGIN` RPATH bundling
(`cmake/promeki_stage_shared_deps.cmake`) like the lame/mpg123 `.so`s.

### 4.3 Cross-platform / ARM
- **`--bit-depth=all` + `--chroma-format=all`** give one runtime-selectable
  library for every depth/subsampling (set per-session via
  `param.i_csp` / `X264_CSP_HIGH_DEPTH`). Avoids the old two-library
  (`libx264` + `libx26410b`) split.
- **x86-64:** needs `nasm` ≥ 2.13 for the asm kernels (already required by
  JPEG-XS; reuse the `find_program(NAMES nasm yasm)` probe). If nasm is
  absent, either fail the flag or pass `--disable-asm` (slower).
- **ARM/aarch64:** x264 ships NEON asm assembled by the C compiler/gas —
  **no nasm needed**. Native ARM builds need no special handling.
- **Cross-compiling** (e.g. x86 host → aarch64): build a
  `_PROMEKI_X264_CROSS_ARGS` list mirroring `_PROMEKI_EXTPROJ_CROSS_ARGS`
  (`CMakeLists.txt:1507`) but in x264's vocabulary —
  `--host=<triple>` and `--cross-prefix=<triple->`. Derive from
  `CMAKE_C_COMPILER_TARGET` / the toolchain file.
- **Default ON** (owner direction). Follows the `PROMEKI_ENABLE_*_DEFAULT`
  probe idiom: on x86 the default can still gate on nasm being found
  (else `--disable-asm`); on ARM no probe needed. Enabling it triggers the
  GPL-compliance notice (§4.5). Add `PROMEKI_USE_SYSTEM_X264` for the
  system-lib path, matching the other `PROMEKI_USE_SYSTEM_*` toggles.

### 4.4 Feature-flag dependency
`PROMEKI_ENABLE_X264` requires `PROMEKI_ENABLE_PROAV` (add the
`FATAL_ERROR` guard next to the JPEGXS one at `CMakeLists.txt:712`).
Generated `config.h` already mirrors `PROMEKI_ENABLE_*` flags.

### 4.5 GPL-compliance notification mechanism (reusable)
**New:** a small CMake helper, placed near the top-level flag resolution
(e.g. a `cmake/PromekiGplComponents.cmake` included from `CMakeLists.txt`),
so the **same mechanism serves libx265 and any future GPL dependency** — a
GPL block just registers itself with one line.

Design:

```cmake
# Any optional-dependency block whose library is GPL-licensed calls this
# once, inside its `if(PROMEKI_ENABLE_<x>)`:
function(promeki_register_gpl_component name license)
    set_property(GLOBAL APPEND PROPERTY PROMEKI_GPL_COMPONENTS "${name} (${license})")
endfunction()

# …in the x264 block:
if(PROMEKI_ENABLE_X264)
    promeki_register_gpl_component("libx264" "GPL-2.0-or-later")
endif()
# …later, the x265 block will add:
#   promeki_register_gpl_component("libx265" "GPL-2.0-or-later")
```

After **all** `PROMEKI_ENABLE_*` flags are resolved, a single emit block:

```cmake
get_property(_gpl GLOBAL PROPERTY PROMEKI_GPL_COMPONENTS)
if(_gpl)
    set(PROMEKI_GPL_BUILD ON CACHE INTERNAL "A GPL-licensed component is enabled")
    list(JOIN _gpl "\n     - " _gpl_list)
    message(WARNING
        "\n"
        "============================================================\n"
        "  GPL COMPLIANCE NOTICE\n"
        "============================================================\n"
        "  One or more GPL-licensed components are ENABLED in this\n"
        "  build of libpromeki:\n"
        "     - ${_gpl_list}\n"
        "\n"
        "  As a result, the resulting libpromeki build is a work\n"
        "  subject to the GNU General Public License, v2.0 or later.\n"
        "  If you distribute it, you must comply with the GPL\n"
        "  (including making complete corresponding source available).\n"
        "  To produce a non-GPL build, reconfigure with the listed\n"
        "  -DPROMEKI_ENABLE_*=OFF flag(s).\n"
        "  See THIRD-PARTY-LICENSES (\"GPL-Licensed Components\").\n"
        "============================================================")
endif()
```

Notes:
- `message(WARNING)` keeps it loud at every configure without failing the
  build (on-by-default must still configure cleanly). The notice repeats on
  reconfigure, which is intentional.
- `PROMEKI_GPL_BUILD` is also mirrored into generated `config.h` so an
  application's "about/licenses" screen can report the GPL posture at
  runtime (optional, cheap).
- Optional future toggle `PROMEKI_ALLOW_GPL` (default ON) could turn the
  warning into a `FATAL_ERROR` for shops that must guarantee GPL-free
  builds in CI — listed as a possibility, not required now.

## 5. The `X264VideoEncoder` backend

**New:** `include/promeki/x264videoencoder.h`, `src/proav/x264videoencoder.cpp`
(modelled on `jpegxsvideocodec.*` for shape and `nvencvideoencoder.cpp` for
the pro-feature plumbing). pImpl hides all `x264.h` types from the header
(`struct Impl; ImplPtr _impl;`).

```cpp
class X264VideoEncoder : public VideoEncoder {
    public:
        X264VideoEncoder();
        ~X264VideoEncoder() override;
        static List<int> supportedInputList();   // the §5.1 table, as PixelFormat IDs
        Error submitFrame(const Frame &) override;
        Frame receiveFrame() override;
        Error flush() override;
        Error reset() override;
        void  requestKeyframe() override;        // sets pic_in.i_type = X264_TYPE_IDR next submit
    protected:
        void  onConfigure(const MediaConfig &) override;
    private:
        struct Impl; using ImplPtr = UniquePtr<Impl>; ImplPtr _impl;
        bool _requestKey = false;
};
```

### 5.1 Input format table (`X264_CSP_*`)
Mirror NVENC's `kFormatTable` but to x264 colour spaces (uses §3.4 geometry):

| PixelFormat | x264 csp | chroma | depth |
|---|---|---|---|
| `YUV8_420_Planar_Rec709` | `X264_CSP_I420` | 1 | 8 |
| `YUV8_422_Planar_Rec709` | `X264_CSP_I422` | 2 | 8 |
| `YUV8_444_Planar_Rec709` | `X264_CSP_I444` | 3 | 8 |
| `YUV10_420_Planar_LE_Rec709` | `I420 \| X264_CSP_HIGH_DEPTH` | 1 | 10 |
| `YUV10_422_Planar_LE_Rec709` | `I422 \| X264_CSP_HIGH_DEPTH` | 2 | 10 |
| `YUV10_444_Planar_LE_Rec709` | `I444 \| X264_CSP_HIGH_DEPTH` | 3 | 10 |
| Rec2020 PQ/HLG 10-bit planar variants | same csp; ColorModel drives VUI | | 10 |

- 10-bit: x264 high-depth expects native-endian `uint16` samples with the
  value right-aligned. Our `_LE_` planar payloads are little-endian 16-bit
  — correct on our LE targets. Document the LE assumption in the table.
- We feed `x264_picture_t.img.plane[c]` / `i_stride[c]` directly from the
  payload planes (no upload/copy — unlike NVENC). Honour each plane's
  `PixelMemLayout::lineStride()` for `i_stride[c]`.
- `supportedInputList()` must mirror this table exactly (the planner picks
  the TPG/CSC output format from it — same contract as JPEG-XS:444-460).

### 5.2 Session setup (`onConfigure` + lazy first-frame init)
Like NVENC, stash config in `onConfigure` and build the real
`x264_t` lazily on first `submitFrame` once dimensions/format are known.

`x264_param_t` setup:
1. `x264_param_default_preset(&p, preset, tune)` — map `VideoPreset`
   (`ultrafast`…`veryslow`) and derive `tune` (`zerolatency` for
   UltraLowLatency/LowLatency; otherwise none/`film`).
2. Dimensions: `i_width`/`i_height`; `i_csp` from §5.1.
3. `i_bitdepth = 8|10` (runtime, thanks to `--bit-depth=all`).
4. Frame rate: `i_fps_num/den` from `MediaConfig::FrameRate`;
   `b_vfr_input = 0`; set `i_timebase_*` for PTS passthrough.
5. Rate control (`MediaConfig::VideoRcMode`):
   - `CBR` → `rc.i_rc_method = X264_RC_ABR`, `i_bitrate`,
     `rc.i_vbv_max_bitrate = i_bitrate`, `rc.i_vbv_buffer_size` set
     (constrained VBV = CBR).
   - `VBR` → `X264_RC_ABR` with `i_bitrate` + optional
     `MaxBitrateKbps`→`i_vbv_max_bitrate`. (Or CRF if a quality key is
     preferred later.)
   - `CQP` → `X264_RC_CQP`, `rc.i_qp_constant = VideoQp`.
6. GOP: `i_keyint_max = GopLength` (or `IdrInterval` if set);
   `b_intra_refresh` optional.
7. B-frames: `i_bframes = BFrames`, `i_bframe_pyramid`. (See §5.4 reorder.)
8. Lookahead: `rc.i_lookahead = LookaheadFrames` when > 0.
9. Threads: `i_threads = 0` (auto) — cross-platform.
10. Annex-B output: `b_annexb = 1` (match NVENC convention; downstream
    MP4/RTP convert via `H264Bitstream`). `b_repeat_headers =
    VideoRepeatHeaders`.
11. VUI colour: write `p.vui.{i_colorprim,i_transfer,i_colmatrix,
    b_fullrange}` from §3.1 `resolveH273()`.
12. Profile/level: `x264_param_apply_profile(&p, profileName)` using §3.3;
    `p.i_level_idc` from parsed level. Auto-select profile from chroma/
    depth when `VideoProfile` empty (4:4:4→`high444`, 4:2:2→`high422`,
    10-bit→`high10`, else `high`) — mirrors `h264ProfileGuid` auto path.
13. Lossless: `VideoEncoderPreset::Lossless` or `VideoQp==0` → `X264_RC_CQP`
    + `i_qp_constant = 0`. x264 supports lossless at any chroma (no 4:4:4
    restriction, unlike NVENC) — note this as a capability advantage.
14. `x264_encoder_open(&p)` → `x264_t*`. Then `x264_encoder_headers()` to
    grab SPS/PPS Annex-B bytes for `Metadata::CodecParameterSets` (cleaner
    than parsing the first IDR; reuse `AvcDecoderConfig`/`H264Bitstream` if
    AVCC form is wanted).

### 5.3 Per-frame submit
1. `selectInputPayload(frame)` → `UncompressedVideoPayload`; map to a
   stack `x264_picture_t pic_in` (`x264_picture_init`, set `img`, `i_pts`
   from payload `pts`, `i_type = X264_TYPE_AUTO` or forced IDR).
2. SEI via §3.2: assemble caption (4), HDR mastering (137) + CLL (144),
   timecode (1, best-effort §6.7) payloads into `pic_in.extra_sei`
   (`x264_sei_payload_t[]`). Allocate payload bytes with `malloc` and set
   `extra_sei.sei_free = free` so x264 reclaims them. HDR sources:
   `idesc.metadata()` (`Metadata::MasteringDisplay`/`ContentLightLevel`)
   falling back to the config values — same precedence as NVENC:547-550.
3. `x264_encoder_encode(h, &nal, &i_nal, &pic_in, &pic_out)`. When
   `i_nal > 0`, a frame is ready → build `CompressedVideoPayload` from the
   concatenated NAL bytes (`pic_out.i_type` → `FrameType`/keyframe flag,
   `pic_out.i_pts`/`i_dts` → payload pts/dts). Hold the source `Frame`
   keyed by pts so `buildOutputFrame` can echo audio/ANC/metadata
   (mirror NVENC's slot→sourceFrame mapping; a `Map<int64_t,Frame>` or a
   small in-flight deque keyed by submission order works).
4. Stamp output metadata (§3.5): `CodecParameterSets`, scan mode,
   duration-from-framerate, GOP position, `FrameType`.

### 5.4 Drain / B-frame reordering / flush
x264 buffers `x264_encoder_delayed_frames(h)` pictures (lookahead +
B-frames). Output emerges in **DTS order with PTS reorder** — handle it at
the `VideoEncoder` level (the `receiveFrame` queue tolerates this; the
payload carries both pts and dts).
- `receiveFrame()` pops from an internal ready-Frame queue (like JPEG-XS's
  `_queue`).
- `flush()`: call `x264_encoder_encode` with `pic_in = NULL` repeatedly
  until `x264_encoder_delayed_frames(h) == 0`, queuing each emitted Frame;
  then enqueue an `EndOfStream`-flagged empty payload (mirror NVENC:736-744).
- **MediaIO note:** `VideoEncoderMediaIO` currently documents a 1-in/1-out
  assumption (no B-frames). For the MediaIO path, **default `BFrames = 0`**
  so each `writeFrame` yields its packet synchronously. B-frames remain
  available through the direct `VideoEncoder` API and become MediaIO-safe
  once the EOS/drain protocol noted in `videoencodermediaio.h:54-57` lands
  (shared with the same NVENC limitation — track together).

### 5.5 Registration
Anonymous-namespace static registrar, identical shape to
`JpegXsVideoCodecRegistrar` (jpegxsvideocodec.cpp:632-657):

```cpp
auto bk = VideoCodec::registerBackend("x264");
VideoEncoder::registerBackend({
    .codecId = VideoCodec::H264,
    .backend = value(bk),
    .weight  = BackendWeight::Vendored,   // vs NVENC "Nvidia" also Vendored — see §8
    .supportedInputs = X264VideoEncoder::supportedInputList(),
    .factory = []() -> VideoEncoder * { return new X264VideoEncoder(); },
});
```

No decoder registration. Whole TU is `#if PROMEKI_ENABLE_X264`.

### 6.6 HDR (mastering-display + CLL)
x264 has no built-in HDR-SEI API, so we **build the RBSP ourselves**
(§3.2 `buildMasteringDisplaySei` / `buildContentLightLevelSei`) and inject
via `extra_sei`. Values come from `MasteringDisplay`/`ContentLightLevel`
(`masteringdisplay.h`/`contentlightlevel.h`) using the same 50000/10000-unit
scaling NVENC's `toNvencMastering`/`toNvencCll` use (nvencvideoencoder.cpp:308-328).

### 6.7 Timecode SEI — caveat
A standalone, spec-valid **pic_timing** SEI with `clock_timestamp`
(SMPTE 12M timecode) depends on VUI/HRD signalling
(`pic_struct_present_flag`, `nal_hrd`/`vcl_hrd`, `cpb_removal_delay`).
NVENC's hardware PTD produces this internally. For x264 we either:
- **(a)** enable x264's own pic-timing where it suffices and let it emit, or
- **(b)** build a minimal pic_timing SEI gated on the VUI flags we set,
  documenting the constraints.

Mark timecode-SEI **best-effort / Phase 2**; do not block 8/10-bit + chroma
+ HDR + captions on it. (Same realism the NVENC plan applies to interlaced.)

### 6.8 Interlaced
`param.b_interlaced = 1` + `param.b_tff` from `VideoScanMode`
(`InterlacedEvenFirst`→tff). x264 does real PAFF/MBAFF (a capability NVENC
lacks here), but field-pair *submission* semantics need validation —
stage behind progressive in testing.

## 8. Backend selection / weight

NVENC ("Nvidia") and x264 will both register `H264` encoders. Unpinned
`VideoCodec("H264")` picks **highest weight**. Options:
- Give x264 a **lower weight** than NVENC (`BackendWeight` has tiers) so
  hardware wins by default when present, and x264 is the portable fallback;
  or
- Leave both `Vendored` and rely on explicit pinning (`H264:x264` /
  `H264:Nvidia`) / `MediaConfig::CodecBackend`.

**Decided (owner direction): hardware before x264.** NVENC (and any future
hardware H.264 backend) must outrank x264 so a GPU box prefers hardware and
headless/ARM boxes transparently fall back to the software encoder. During
implementation, confirm the `BackendWeight` tiers (`backendweight.h`) express
a hardware > software ordering; if the current tiers don't, register x264 at
a strictly lower weight than NVENC (or add a `Software`/`Fallback` tier below
`Vendored`). Explicit pinning (`H264:x264` / `H264:Nvidia`) and
`MediaConfig::CodecBackend` always override the weight.

     `frameMbsOnly`; the test confirms interlaced clears
     `frame_mbs_only_flag` and progressive sets it.

     implemented in-band.** x264 owns the pic_timing SEI
     (`x264_sei_pic_timing_write`) and **hardcodes `clock_timestamp_flag=0`**,
     so it emits `pic_struct` (used above for interlaced) but cannot carry
     SMPTE `clock_timestamp`; a hand-injected pic_timing would either
     duplicate x264's (malformed) or, without `b_pic_struct`, lack the
     required `pic_struct_present_flag`. Proper in-band timecode needs an
     x264 patch or a bitstream post-pass (out of scope). Instead, **Timecode

     JPEG-XS) — carried losslessly to the matching decoder. Test confirms
     the round-trip.

     the MediaIO 1-in/1-out → N-in/M-out EOS/drain protocol change noted at

     limitation**. B-frames already work through the direct `VideoEncoder`
     API (drain/flush handles reorder); making them MediaIO-safe should be a
     focused cross-cutting task that lands the shared protocol for both
     backends at once, rather than reshaping the MediaIO contract from
     inside the x264 backend. Tracked as its own item.

## 10. Open questions / risks

- **Timecode SEI** standalone validity without full HRD (§6.7) — accept
  best-effort or invest in proper pic_timing/HRD.
- **B-frames through MediaIO** share the deferred EOS protocol noted for
  NVENC (`videoencodermediaio.h:54-57`); default `BFrames=0` until then.
- **Backend-weight tiering** (§8) — does `BackendWeight` already express
  hardware-vs-software, or do we add a tier?
- **Reference decoder for functest** on non-NVIDIA / ARM CI — pick one
  (system ffmpeg `h264` for test-only validation, kept out of the shipped
  library).
